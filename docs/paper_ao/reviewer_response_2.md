## Q1. Full formal definitions of δ(C) and γ(C), including how γ aggregates concurrent validations and how ties/conflicts are handled.

**Response.** We have provided complete formal definitions in §4.2 (status derivation) and §4.5 (confidence aggregation). The three γ variants are:

- **γ_default**: G-Counter max over all VALIDATE events (O(|V|) scan)
- **γ_agg**: Per-actor maxima with cross-validation bonus (β = 0.05 per distinct validator)
- **γ_cal**: Accuracy-weighted mean with calibration profiles

**Concurrent validation handling.** When multiple VALIDATE events arrive concurrently, they are ordered by the total order ≺ (timestamp primary, event_id lexicographic tiebreaker). The γ functions are commutative over the event set: γ_default takes the max regardless of order; γ_agg computes per-actor maxima and aggregates; γ_cal computes weighted means. All three are deterministic functions of the event set, not the linearization order (Theorem 9, §4.8).

**Ties and conflicts.** For γ_default, if two validators report the same confidence, the max is unchanged. For γ_agg, duplicate validators do not increase the bonus (N_vals counts distinct actors). For γ_cal, conflicting validations (e.g., agent_1 reports 0.9, agent_2 reports 0.5) are weighted by accuracy scores, so high-accuracy validators dominate. The LUB status semantics guarantees that a DEPRECATE event always dominates a VALIDATE event in status, but confidence remains the maximum of all validations (G-Counter monotonicity, Theorem 5).

## Q2. How exactly were the 201 EventChains derived from the AML HI-Small dataset? Describe the mapping and release scripts.

**Response.** The mapping methodology is fully described in Appendix D ("Reproducibility and Experiment Runner"), which we have expanded. The key points:

**Data source.** The IBM AML HI-Small dataset contains 60 distinct account records with transaction-level features. Each account was treated as a distinct "capability" (a financial-behavior pattern) and mapped to one EventChain.

**Mapping pipeline.**
1. **L1 front matter:** Each account's ID became the concept_id; the account's features (balance, transaction count, anomaly score) were serialized as YAML metadata.
2. **L2 body:** The account's transaction history was summarized in Markdown prose (a synthetic narrative generated by template, since the raw data is tabular).
3. **Event injection:** Governance events were synthetically injected to simulate a multi-agent lifecycle workflow:
   - Every chain: at least one REGISTER (genesis) + one VALIDATE
   - 10% of validated chains: DEPRECATE (simulating obsolete capability)
   - 5% of validated chains: FORK (simulating competing interpretation)
   - EVIDENCE and RELATE events interleaved at random indices
   - Confidence values sampled uniformly from [0.50, 0.95] with ~5% at 1.0

**Scripts.** The reproduction package includes `scripts/reproduce/reproduce_e6.sh`, which downloads the AML dataset, runs the DataImporter pipeline, and injects governance events. The mapping is 96.8% real data (REGISTER events from account records) and 3.2% synthetic governance events.

**Validation of realism.** The uniform confidence distribution is a conservative lower bound; real experts would show higher variance. The 60/20/10/6.7/3.3 split is inspired by open-source governance patterns, not AML domain data. Domain-level evaluation with human AML experts (E5) is in progress.

## Q4. Detail the event alphabet partition (Σ_life vs Σ_comm) and list implemented action types with preconditions. How are preconditions evaluated under concurrency?

**Response.** The event alphabet is fully formalized in §4.1:

```
Σ_life = {REGISTER, VALIDATE, DEPRECATE, FORK, ARCHIVE}
Σ_comm = {RELATE, EVIDENCE, SEAL, ANNOUNCE, PUBLISH, SNAPSHOT, CALIBRATE, SYNC_DASHBOARD, LISTEN, REVOKE}
```

**Action preconditions** (from adl_core_ontology.yaml, §4.6):
- VALIDATE: status EQ provisional, confidence GTE 0.5, validator_count GTE 1
- DEPRECATE: status EQ validated
- FORK: status IN {provisional, validated}
- ARCHIVE: status IN {provisional, validated, forked, deprecated}
- RELATE: (no preconditions; communication event)
- REVOKE: (no preconditions; communication event)

**Precondition evaluation under concurrency.** Preconditions are evaluated against a derived snapshot (δ(C), γ(C)) computed from the chain at the time of append. Under concurrent edits, each agent computes the snapshot from its local view of the chain. If two agents append simultaneously:
1. Both compute preconditions against the same pre-append snapshot
2. Both append if preconditions are satisfied
3. The resulting events are ordered by ≺ (timestamp + event_id tiebreaker)
4. The final state is the LUB of all post-append events (CRDT semantics)

This is the standard CRDT convergence pattern: preconditions are evaluated locally, and the merge resolves conflicts deterministically. Theorem 9 (§4.8) proves that the merged state is unique and independent of merge order.

## Q5. Precise partial order over status and confidence for merging. How do you handle divergent validations or revocations arriving in different orders?

**Response.** The status lattice is a total order: provisional < forked < validated < deprecated < archived. The LUB is the maximum element. Confidence is a G-Counter (max over all VALIDATE events). Theorem 9 (§4.8) proves commutativity, associativity, and idempotence.

**Divergent validations.** If agent_1 appends VALIDATE (confidence 0.9) and agent_2 appends VALIDATE (confidence 0.5) concurrently, the merged chain contains both events. γ_default = max(0.9, 0.5) = 0.9. γ_agg = per-actor maxima + bonus. γ_cal = accuracy-weighted mean. All three are independent of merge order because they are functions of the event set, not the sequence.

**Divergent revocations.** A REVOKE event is a communication event (not in Σ_life), so it does not affect δ(C). The Revoked predicate (§3.4.4) checks only whether a REVOKE event exists, regardless of order. If one branch revokes a relation and another does not, the merged chain contains the REVOKE event, so the relation is revoked in the final state. This is monotonic: revocation, once recorded, cannot be undone (append-only).

**Conflicting lifecycle events.** If one branch appends VALIDATE and another appends DEPRECATE concurrently, the merged chain contains both. The LUB status is deprecated (since deprecated > validated). The confidence remains the max of all validations (G-Counter). This is semantically stronger than last-event-wins: it guarantees status never regresses.

## Q6. Why is HoldsAt independent of confidence? Would a thresholded HoldsAt_θ variant be practical?

**Response.** We have added a dedicated explanation in §3.4.4 (immediately after the revocation semantics). The core rationale is the ontological distinction between existence and epistemic strength:

- **HoldsAt** checks existential conditions: does the relator exist at time t? This is binary (yes/no) and determined by the presence/absence of a REVOKE event.
- **Confidence** is epistemic strength: how confident are we in the relation? This is graded [0,1] and determined by the RELATE event's payload.

A relation with confidence 0.3 still holds; a relation with a REVOKE event does not hold, regardless of the original confidence. This separation prevents conflating "we are uncertain about this relation" (low confidence) with "this relation is terminated" (revocation).

**Thresholded variant.** We have defined an optional consumer-side filter: HoldsAt_θ = HoldsAt ∧ confidence ≥ θ. This is not a core ontology axiom but a query-level filter that downstream systems can apply. It does not affect the event-first derivation semantics.

## Q7. Plan for OWL/SHACL encodings of the two-level account and identity/dependence axioms. Which fragments are representable?

**Response.** We have completely rewritten Appendix A to address this. The OWL 2 DL fragment includes:

- Core category axioms (adl:Event ⊑ bfo:occurrent, adl:Concept ⊑ bfo:GDC, etc.)
- L3 relation axioms (isomorphic-to, specialisation-of, fork-of, mitigated-by) as object properties with domain/range constraints
- Five competency questions (CQ1–CQ5) with answers

**Intentionally outside OWL 2 DL:**
1. Temporal reasoning (HoldsAt, Revoked conditions)
2. Derived state (δ, γ aggregation over sequences)
3. Preconditions (Comparator rules with lambda dispatch)
4. Cryptographic integrity (SHA-256 correctness)

**SHACL usage.** SHACL shapes govern L1 schema conformance (adl_type enumeration) and L2 template structure ([Observation], [Reasoning], [Conclusion] sections). SHACL is not used for L3 relation validation or L4 action preconditions, which are handled by the ActionExecutor. Full SHACL integration is future work (FW6).

**Migration paths.**
- To nanopublications: ADL Lite events export to RDF-star quoted triples (Appendix F). Nanopublication assertions can be imported as RELATE events with SHA-256 content-addressed references.
- To PROV-O + Linked Data Proofs: The PROV-O mapping is already implemented (§6.7). Events map to prov:Activity, actors to prov:Agent, chains to prov:Entity. Linked Data Proofs (Ed25519 signatures) are implemented in Phase 1.5.
- To Git-native: ADL Lite EventChains are stored as Markdown files in standard Git repositories; commit signatures provide repository-level integrity, event hashes provide concept-level integrity.

## Q9. The paper mentions SSA in L2; can you define it and clarify how it interacts with lifecycle governance and validation policies?

**Response.** We have expanded the SSA definition in §4.1 ("Document Model and Event Alphabet") to include its interaction with lifecycle governance:

**Definition (SSA).** The Singular Subjectivity Assertion is a document-level constraint stating that every L2 Markdown body is authored by a single actor at a single time. It is enforced by the parser, not the ActionExecutor.

**Interaction with lifecycle governance:**
1. A concept in provisional status may contain SSA-violating prose (it is unvalidated).
2. A VALIDATE action's precondition requires that the L2 body pass SSA checking, ensuring validated concepts have traceable narrative provenance.
3. FORK events inherit the parent's L2 body but must satisfy SSA for the new actor's rationale.

**Interaction with validation policies.** The SSA prevents "ghost-authored" reasoning that cannot be traced to an event. Contradictory subjective assertions must be recorded as separate L4 events (EVIDENCE or RELATE) by different actors. Full machine-checkable enforcement is planned via SHACL shape validation (FW6).
