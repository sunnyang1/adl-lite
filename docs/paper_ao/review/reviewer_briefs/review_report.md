## 1. 原创性（Originality）

**评分：7 / 10**

**优点：**
- 事件优先（event-first）的立场在能力治理领域是新颖的，填补了现有治理层（KYA 权限、AgentSafe 架构安全）未能覆盖的能力生命周期注册表空白
- 将能力建模为派生自事件链的通用依赖持续体（GDC），并在三个上层本体（BFO、DOLCE、UFO）之间建立映射，具有应用本体学的理论价值
- 将密码学完整性、确定性生命周期推导和声明式前置条件集成到轻量级 Markdown 原生工具包中的设计具有工程独创性

**问题与建议：**

- 🔴 **Major** — 上层本体映射中的概念近似性问题
  - **Actor → DOLCE `physical_agent`**：LLM 软件代理被映射为 DOLCE 的 `physical_agent`，但 DOLCE 的 physical agent 通常指人类或具身实体。论文将此标记为“近似”是诚实的，但 Applied Ontology 审稿人可能会质疑这种近似是否足以支撑跨本体推理。建议：在 §3.2 增加脚注说明，指出这是一个设计权衡（因为 DOLCE 缺乏明确的“软件代理”类别），并讨论未来映射到 UFO `Agent` 的替代方案。
  - **EventChain → BFO `process`**：EventChain 作为记录（序列化的 ICE）与作为过程（时间中展开的事件序列）之间存在歧义。论文在 §3.2.4 的两层本体论中区分了 `EventChain_process` 和 `EventChain_record`，但 Table 2 中的映射仅标注为 `process`，这可能导致误导。建议：在 Table 2 的 Note 列中明确标注“两层区分见 §3.2.4”。

- 🔴 **Major** — OntoClean 评估中的依赖类型混淆
  - 论文在 §3.3 声称 Concept 满足“刚性存在依赖”（rigid existential dependence）于 EventChain 记录。但 §3.4 又说 Concept 对 EventChain 记录是“通用依赖”（generic dependence）。根据 Fine (1995) 的区分，刚性依赖要求概念 *不能* 存在于该 *特定* 承载者之外，而通用依赖要求概念依赖于 *某类* 承载者。这两个概念在形式化上不可互换。建议：统一使用 **通用依赖** 来描述 Concept → EventChain 记录的关系（因为概念可以存在于多个副本中），并明确说明“刚性”仅适用于 genesis hash 作为身份标准，而非存在依赖关系。

- 🟡 **Minor** — OWL 2 DL 片段的充分性不足
  - §3.5 和 Appendix A 中的 OWL 2 DL 片段仅声明了 4 个类包含关系和 2 个数据类型属性。论文承认该片段“不捕获 δ(C) 或 γ(C)”，而这两个推导函数是系统的核心操作语义。对于 Applied Ontology 审稿人，该片段可能看起来过于单薄。建议：在 §3.5 中明确说明该片段的定位是“概念对齐的锚点，而非完整操作语义的可执行编码”，并将完整操作语义编码明确标记为 FW1（未来工作）。

- 🟡 **Minor** — “Horn-clause 片段”声明可能属于范畴错误
  - §3.6 和 §4.5 声称前置条件语言是“Horn-clause 片段的 FOL”。但标准 Horn-clause 逻辑包含头谓词和体谓词之间的蕴含关系，而 ADL Lite 的前置条件仅是地面比较的合取（无蕴含）。建议：将此声明修改为“每个前置条件规则是一个地面原子比较，其合取等价于一个无变量的 Horn-clause 合取，且评估在 P 时间内可判定”，从而准确反映其表达力限制。

## 2. 方法论（Methodology）

**评分：6 / 10**

**优点：**
- 形式化推导语义（事件字母表、良构性谓词 WF(C)、δ(C) 和 γ(C)）是完整的，定义清晰，自包含
- 六个核心定理（确定性、分叉合流、过渡单调性、置信度有界性、置信度单调性、状态-置信度一致性）覆盖了 EventChain 的关键计算性质
- 前置条件语言采用闭 Comparator 枚举（无 eval()），从实现角度确保了安全性和可判定性
- 对抗性试验（7 个可检测 + 4 个不可检测攻击场景）对 Phase 1 威胁模型边界进行了诚实评估

**问题与建议：**

- 🔴 **Major** — 定理证明未机器检验，且未建立机器检验路线图
  - 论文承认七个定理的证明是“严格自然语言论证”而非机器检验（Coq/TLA+）。对于 Applied Ontology 审稿人，这本身不一定构成问题——但论文声称这些定理是“已证明的”（proved），而附录 E 中的证明实际上是自然语言论证。建议：在 §4.4 将定理的陈述方式从“Theorem X (Property). Proof: ...”调整为“Theorem X (Property). Proof Sketch: ...”，以明确这些是证明梗概而非完整证明。同时，在 FW10 中明确承诺具体的机器检验目标（例如：在 TLA+ 中验证 Theorem 1-3 的 TLC 模型已覆盖长度 ≤20 的序列，并计划在 Coq 中完成无界链的证明）。

- 🔴 **Major** — E19 完全为作者估算，无实测数据
  - Table 8（E19）中的 LOC、延迟、错误数均为“作者估算”（作者标注）。这降低了作为“实证基准”的说服力。建议：要么（1）将此表重新定位为“概念性设计比较”而非“实证基准”，并明确说明其方法论性质；要么（2）在附录 L 中补充至少一个实际任务的测量实现（例如，用 ADL Lite 和 nanopub 脚本分别实现相同的治理任务，记录实际 LOC 和运行时间）。目前的 E19 作为“设计比较”有价值，但作为“基准”过于薄弱。

- 🟡 **Minor** — Theorem 8 的 O(1) 假设未明确说明
  - Theorem 8（前置条件评估复杂度）声称 O(1) 时间，但前提假设是“派生快照已缓存”。该假设在定理陈述中未明确标注。建议：在定理陈述中显式添加条件“(assuming cached derived snapshot)”或在脚注中说明。

- 🟡 **Minor** — 比较基线未在相同硬件上运行
  - Table 7（E12 比较基准）中 ADL Lite 的测量在 Apple M2 上完成，而 nanopub 和 PROV-O 的数值来自已发表文献（不同硬件）。脚注已承认此限制，但审稿人可能仍质疑比较的公平性。建议：在表注中补充更详细的解释，说明“由于 nanopub 和 PROV-O 的实现依赖其各自的生态（RDF 三元存储、SPARQL 端点），跨硬件的直接控制实验在 Phase 1 中不可行；FW11 计划进行 head-to-head 基准测试”。

- 🟡 **Minor** — E6b 的扩展声明是线性外推，非实测
  - E6b 将 9,300 事件的结果外推到 1M 事件（47.7 秒）。论文已诚实标注为“线性外推，非实测验证”，但这仍可能被质疑。建议：在 §5.5 中增加一句话说明为何外推是合理的（例如：事件处理是 O(n) 的独立哈希计算，无 superlinear 数据结构），并指出 Phase 2 的 ARCHIVE 事件和冷存储机制将解决潜在的线性增长问题。

- 🟡 **Minor** — “非共谋”假设未形式化
  - Theorem 5（置信度单调性）要求“来自新的非共谋参与者”的验证事件。但“非共谋”在论文中未定义，也未作为可验证的操作假设。建议：在 §4.4.5 的 Lemma 1（共谋漏洞上界）之后，明确说明 Theorem 5 的“非共谋”假设是 Phase 1 的协作审计模型的设计假设，而非形式化证明中的公理；在 Phase 3 中，该假设将通过 Ed25519 签名和 DIDs 替换为密码学验证。

## 3. 结果（Results）

**评分：7 / 10**

**优点：**
- E1（链完整性）、E2（状态推导）、E3（快照一致性）、E4（前置条件执行）均达到完美得分（P = R = F1 = 1.0），验证了架构正确性
- E6 在 201 条链（9,300 事件）上零完整性失败，吞吐量 20,955 事件/秒，线性扩展
- E14（共谋漏洞）和 E15（防御深度缺口）的负面结果被诚实报告，增强了可信度
- 对抗性试验（7 个可检测攻击场景 × 201 链 = 1,407 次试验，100% 检测率）对完整性验证机制进行了充分的压力测试

**问题与建议：**

- 🔴 **Major** — 缺乏领域级评估（E5 未执行）
  - 论文的定位是“框架论文”，但审稿人可能质疑：一个能力治理框架是否在没有领域验证的情况下具有足够的实证价值？E5（AML 专家评分）被标记为“初步设计；未执行”。建议：在 §5.7 中明确说明框架论文的定位——“本文的目标是建立本体论和形式基础，而非验证领域有效性；E5 已设计但将在后续工作中执行”。同时，考虑在 §5.4 的 AML 案例研究中增加一个更具体的“治理工作流程”示例，展示 AML 模式如何通过 REGISTER → VALIDATE → EVIDENCE → DEPRECATE 的生命周期被治理，从而在不执行 E5 的情况下增强领域可信度。

- 🔴 **Major** — 仅使用单一硬件平台（Apple M2）
  - 所有性能测量均在 Apple M2（ARM、macOS）上完成。虽然对于架构验证已足够，但审稿人可能质疑结果的可移植性。建议：在 §5.10 或附录 D 中增加说明，指出 CI 已在 Python 3.10/3.11/3.12 的矩阵上测试（GitHub Actions 的 Linux 运行器），但详细的性能基准仅在一台机器上完成，因为论文关注相对复杂度而非绝对性能。

- 🟡 **Minor** — E17（多智能体模拟）的 89.7% 有效性是模拟结果
  - E17 使用 5 个智能体、3 个概念、20 轮的前置条件 ON/OFF 模拟，得到 89.7% 的坏过渡预防率。论文明确说明这是“模拟”而非真实智能体评估，但审稿人可能仍质疑其外部效度。建议：在 §5.8 中增加一句话，说明该模拟的目的是验证前置条件系统的机制有效性（mechanism validation），而非预测真实智能体行为，并指出 E20 计划使用真实 LLM 智能体进行验证。

- 🟡 **Minor** — Pydantic 开销占 58.4% 但未验证
  - E6 声称 Pydantic 物化占 58.4% 的延迟，但未说明如何测量。建议：在附录 D 中增加一句话说明测量方法（例如：通过 cProfile 或 line_profiler 进行 CPU 热点分析），或提供可复现的剖析脚本。

- 🟡 **Minor** — 2,204 个 E2 测试用例的推导不透明
  - E2 声称测试了“长度不超过 3 的所有事件序列”，但论文未说明 2,204 这个数字的来源。对于 5 个生命周期事件类型，长度 3 的序列数应为 5³ = 125，加上 7 个边界情况 = 132。2,204 可能是包含通信事件或全排列的更大组合。建议：在 §5.2 或附录 D 中增加一句话解释 2,204 的推导方法（例如：是否包含全排列、组合事件、或多种初始状态？）。

## 给作者的整体建议

ADL Lite 的核心贡献是扎实的：一个将事件优先本体论承诺转化为可部署、Markdown 原生、密码学增强的能力生命周期注册表的框架。论文的本体分析（§3）和形式推导语义（§4）构成了有力的理论基础，而架构验证（§5）在正确性验证方面达到了完美得分。论文的诚实性（尤其是在威胁模型、局限性和负面结果方面）是其最大的竞争优势。

然而，论文面临三个主要挑战需要在修改中解决：

**第一，术语精度。** 本体论分析中的“刚性存在依赖”与“通用依赖”之间的术语混淆是 Applied Ontology 审稿人最可能标记的问题。这不是一个严重的概念错误，而是一个术语不一致问题，需要明确的澄清和统一。一旦修正，OntoClean 评估的说服力将显著增强。

**第二，证明方法的诚实标签。** 将自然语言论证标记为“Proof:”而非“Proof Sketch:”在形式化审稿人看来可能显得过于自信。Applied Ontology 期刊接受自然语言证明，但期望作者明确标记其为证明梗概。这是一个简单的标签修改，但能显著提升形式化审稿人的信任度。

**第三，实验结果的外部效度。** E5 未执行和 E19 完全为估算这两个问题需要在论文中重新定位。将论文明确定位为“框架论文”（建立本体论和形式基础，而非领域验证），并将 E19 重新定位为“概念性设计比较”而非“实证基准”，是合理的策略。审稿人会接受这种定位，前提是论文明确说明这些结果的方法论性质。

总的来说，这是一篇值得发表的论文，其修改工作量可控。修改后，我预计其通过 Applied Ontology 或 ESWC/ISWC 的评审概率较高（推荐：Minor Revision → Accept）。
