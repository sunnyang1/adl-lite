\subsection{Document Model: L1--L4 as Conceptual Modeling Layers}\label{subsec:document-model}

ADL Lite organizes capability documents into four semantic layers, each with distinct syntax, role, and corresponding event types. The layering separates identity metadata (L1), narrative content (L2), typed assertions (L3), and executable actions (L4), enabling both human readability and machine processing within a single Markdown file. Table~\ref{tab:document-model} summarizes the document model.

\begin{table}[htbp]
\centering
\caption{The L1--L4 Document Model. Each layer contributes distinct semantics to the capability document.}
\label{tab:document-model}
\begin{tabular}{@{}llll@{}}
\toprule
\textbf{Layer} & \textbf{Syntax} & \textbf{Role} & \textbf{Event Type} \\
\midrule
L1 & YAML front matter & Identity metadata (derived snapshot) & SNAPSHOT \\
L2 & Markdown body & Human/LLM narrative prose & --- \\
L3 & \texttt{adl:relation}, \texttt{adl:evidence}, \texttt{adl:seal} & Typed semantic assertions & RELATE, EVIDENCE, SEAL \\
L4 & \texttt{adl:action} & Typed actions with preconditions & REGISTER, VALIDATE, DEPRECATE, \ldots \\
\bottomrule
\end{tabular}
\end{table}

The L1 layer comprises YAML front matter containing identity metadata; critically, these fields are \emph{derived snapshots} recomputed from the EventChain. The L2 layer consists of Markdown body prose subject to the Singular Subjectivity Assertion (SSA).

\textbf{Definition (SSA).} The Singular Subjectivity Assertion is a document-level constraint stating that every L2 Markdown body is authored by a single actor at a single time. It is enforced not by the ActionExecutor (which governs L4 events), but by the parser: any L2 section containing multiple contradictory subjective claims without explicit attribution is flagged as an SSA violation. The SSA ensures that the L2 narrative layer remains traceable to a single source of subjectivity, preventing ``ghost-authored'' reasoning that cannot be traced to an event. Contradictory subjective assertions must be recorded as separate L4 events (e.g., \textsc{Evidence} or \textsc{Relate}) by different actors. Full machine-checkable enforcement of SSA is planned via SHACL shape validation (FW6).

The L3 layer introduces typed semantic assertions through fenced blocks supporting predicates such as \texttt{isomorphic-to}, \texttt{specialisation-of}, and \texttt{related-to}, with lifecycle operations (establishment, modification, revocation) recorded as \texttt{RELATE} events. These predicates serve a dual purpose: they express domain relationships between concepts, and they provide a mechanism for linking near-duplicate concepts that may arise from multi-agent authoring. For example, if two agents independently register ``gradient-explosion'' and ``exploding-gradients'', the recommended workflow is to detect the near-duplicate by string similarity and link them with the strongest applicable predicate (\texttt{isomorphic-to} for structural equivalence). If one concept is strictly superior, the other is marked as \texttt{deprecated} with a \texttt{FORK} or \texttt{RELATE} event pointing to the canonical concept. Automated canonicalisation via embedding clustering is deferred to future work (FW7).

\textbf{Invariant 2 (L3 Relation Reconciliation).} When either the source or target concept of a \texttt{RELATE} event undergoes a state transition to \texttt{deprecated} or \texttt{archived}, the relation's validity is determined by the status of both endpoint concepts. A relation is valid only if at least one endpoint is in \texttt{validated} status and neither endpoint is in \texttt{archived} or \texttt{deprecated} status. Let $r$ be a relation with predicate $p$ from concept $C_1$ to concept $C_2$, and let $S(C)$ denote the current lifecycle status of concept $C$. Then:
\begin{align}
    \text{valid}(r) \iff S(C_1) \notin \{\text{archived}, \text{deprecated}\} \land S(C_2) \notin \{\text{archived}, \text{deprecated}\} \land \bigl(S(C_1) = \text{validated} \lor S(C_2) = \text{validated}\bigr).
\end{align}
Relations with at least one endpoint in \texttt{validated} status and no endpoint in \texttt{archived} or \texttt{deprecated} status remain valid for query and traversal; relations between two \texttt{provisional} concepts, or where either endpoint is \texttt{deprecated} or \texttt{archived}, are excluded from the warm index. If a forked concept $C'$ is created via a \texttt{FORK} event from $C$, the parent concept retains all existing relations, while $C'$ inherits \texttt{isomorphic-to} and \texttt{specialisation-of} links from $C$ but not \texttt{analogical-to} links, which are re-evaluated during the forking agent's validation workflow. This invariant ensures that the relation graph remains navigable during multi-agent reorganisation and that stale or superseded edges do not pollute semantic queries.

\subsection{EventChain: Core Data Structure}\label{subsec:eventchain}

The EventChain-record (the serialized information content entity, see \S\ref{subsec:two-level-account}) is the fundamental data structure of ADL Lite: an append-only sequence of cryptographically linked events. An event comprises ten fields: \texttt{event\_id}, \texttt{concept\_id}, \texttt{event\_type}, \texttt{actor}, \texttt{reasoning}, \texttt{timestamp}, \texttt{payload}, \texttt{previous\_event\_id}, \texttt{hash}, and an optional \texttt{signature} (for Ed25519 signature verification, \S\ref{subsec:trust-model}). The event type system is partitioned into lifecycle events $\Sigma_{\text{life}}$ and communication events $\Sigma_{\text{comm}}$:

\begin{align}
    \Sigma_{\text{life}} &= \{\text{REGISTER},\; \text{VALIDATE},\; \text{DEPRECATE},\; \text{FORK},\; \text{ARCHIVE}\} \\
    \Sigma_{\text{comm}} &= \{\text{RELATE},\; \text{EVIDENCE},\; \text{SEAL},\; \text{ANNOUNCE},\; \text{PUBLISH},\; \text{SNAPSHOT}, \nonumber \\
    &\qquad\; \text{CALIBRATE},\; \text{SYNC\_DASHBOARD},\; \text{LISTEN}\}
\end{align}

\subsection{Canonical Serialization and Cryptographic Integrity}\label{subsec:crypto}

Each event's hash is computed from canonical JSON serialization with lexicographically sorted keys, six-decimal rounding, and the timestamp. The canonicalization policy is version-locked in the codebase: a schema version identifier (e.g., \texttt{canon-v1}) is included in the hash input, ensuring that future changes to rounding precision or key ordering do not retroactively invalidate existing hashes. If a new canonicalization version is introduced, old chains remain valid under their original version, while new chains adopt the updated policy. This prevents identity drift when serialization conventions evolve. Integrity verification $\text{VerifyIntegrity}(C)$ checks four conditions for a chain $C = [e_1, \ldots, e_n]$. These are: (1)~genesis anchoring ($e_1.\text{previous\_event\_id} = \text{null}$ unless an ARCHIVE event is present), (2)~cryptographic linkage ($e_i.\text{prev\_hash} = e_{i-1}.\text{hash}$), (3)~hash correctness ($e_i.\text{hash} = \text{SHA-256}(\text{Canon}(e_i))$), and (4)~no dangling \texttt{previous\_event\_id} on the genesis event. When $\text{full}=\text{true}$, it additionally verifies archive file hashes against stored \texttt{archive\_pointer} values.

\subsection{Precondition Language and Action Executor}\label{subsec:precondition-executor}

\subsubsection{Formal Precondition Language}\label{subsubsec:precondition-language}

The ActionExecutor validates L4 action blocks against preconditions expressed in a constrained, decidable language. A precondition rule is a triple $\langle \text{field}, \text{comparator}, \text{value} \rangle$ with comparator alphabet $\mathcal{K} = \{\text{EQ}, \text{NEQ}, \text{GT}, \text{GTE}, \text{LT}, \text{LTE}, \text{IN}, \text{EXISTS}\}$. Table~\ref{tab:comparator-semantics} gives the formal semantics.

\begin{table}[htbp]
\centering
\caption{Comparator semantics: explicit operator mapping to ground predicates.}
\label{tab:comparator-semantics}
\begin{tabular}{@{}lll@{}}
\toprule
\textbf{Comparator} & \textbf{Predicate} & \textbf{FOL encoding} \\
\midrule
EQ & Equality & $x = y$ \\
NEQ & Disequality & $\neg(x = y)$ \\
GT & Greater than & $x > y$ \\
GTE & Greater than or equal & $x \geq y$ \\
LT & Less than & $x < y$ \\
LTE & Less than or equal & $x \leq y$ \\

\bottomrule
\end{tabular}
\end{table}

\textbf{BNF grammar.} The complete syntax of the precondition language is:
\begin{align*}
    \text{rule} &::= \langle \text{field}, \text{comparator}, \text{value} \rangle \\
    \text{field} &::= \text{identifier} \quad (\text{e.g., } \texttt{status}, \texttt{confidence}, \texttt{actor}) \\
    \text{comparator} &::= \text{EQ} \mid \text{NEQ} \mid \text{GT} \mid \text{GTE} \mid \text{LT} \mid \text{LTE} \mid \text{IN} \mid \text{EXISTS} \\
    \text{value} &::= \text{scalar} \mid \text{set} \mid \text{null} \\
    \text{rule\_list} &::= \text{rule} \mid \text{rule\_list} \; \text{AND} \; \text{rule\_list}
\end{align*}

\textbf{Formal semantics.} Let $\mathcal{D}$ be the domain of values (scalars, sets, and $\text{null}$). Let $\text{Snapshot}(C)$ be the derived snapshot of chain $C$, a finite map from field names to $\mathcal{D}$. The semantic function $\text{eval}(r, C) \in \{\text{true}, \text{false}\}$ is defined for a rule $r = \langle f, \kappa, v \rangle$ by:
\begin{equation}
    \text{eval}(\langle f, \kappa, v \rangle, C) = \text{apply}(\kappa, \text{lookup}(f, C), v)
\end{equation}
where $\text{lookup}(f, C) = \text{Snapshot}(C)[f]$ (the value of field $f$ in the snapshot, or $\text{null}$ if absent), and $\text{apply}(\kappa, x, y)$ is the comparator-specific predicate from Table~\ref{tab:comparator-semantics}. The snapshot is pre-computed in $\mathcal{O}(n)$ time by a single scan of $C$; subsequent rule evaluations are $\mathcal{O}(1)$.

\textbf{Composition semantics.} A rule list $R = [r_1, \ldots, r_k]$ is evaluated as a conjunction: $\text{eval}(R, C) = \bigwedge_{i=1}^{k} \text{eval}(r_i, C)$. Evaluation is short-circuit: if $\text{eval}(r_j, C) = \text{false}$ for any $j < k$, the remaining rules are not evaluated. The language is a variable-free ground fragment: each rule is a single atomic comparison over a cached derived snapshot, with no quantification, no recursion, and no dynamic code execution. The conjunction of $k$ rules is a variable-free conjunction of ground atoms, evaluated in $\mathcal{O}(k)$ time (Theorem~8).

Precondition evaluation is pure and referentially transparent: $\text{eval}(r, C)$ uses a cached derived snapshot, where $\text{lookup}(f, C) = \text{Snapshot}(C)[f]$. The comparator dispatch is implemented via explicit lambda mapping without dynamic code execution.

\textbf{Theorem~8 (Precondition Evaluation Termination and Complexity).} For any well-formed EventChain $C$ and any precondition rule $r$, $\text{eval}(r, C)$ terminates in $\mathcal{O}(1)$ time (assuming the derived snapshot is cached). For a list of $k$ rules, total time is $\mathcal{O}(k)$ and space is $\mathcal{O}(1)$. \emph{Proof Sketch:} Both field lookup and comparator dispatch are $\mathcal{O}(1)$; the language contains no recursion or quantification. See Appendix~E for the full argument.

Table~\ref{tab:complexity} summarizes the complexity; the decidability and complexity-class discussion (membership in \textbf{P}) is in Appendix~H.

\subsubsection{Action Executor and Consensus Engine}\label{subsubsec:action-consensus}

The ActionExecutor validates L4 action blocks against a closed action registry defined in \texttt{adl\_core\_ontology.yaml}. Each action definition specifies preconditions as a list of \texttt{PreconditionRule} objects. The ConsensusEngine governs the resulting status transitions by appending events to the chain; the final status is derived via the CRDT LUB over all lifecycle events (Theorem~1), not by a transition matrix. Forking creates a new child chain with a fresh \texttt{REGISTER} event while the original chain's status is the LUB of its previous status and \texttt{forked}; both evolve independently.

\subsection{Formal Derivation Semantics}\label{subsec:formal-semantics}

This section provides the formal foundation for the derivation semantics. All definitions are self-contained; expanded proof sketches and sensitivity analysis appear in Appendix~E. We have drafted a TLA$^{+}$ specification of the EventChain state machine for lengths up to 20 events (FW10); extension to 100--1000 events is planned for the next revision. We establish Theorems~1--3 through rigorous natural-language argument in this paper.

A chain $C = [e_1, \ldots, e_n]$ is well-formed, denoted $\text{WF}(C)$, if it satisfies 12 axioms (genesis anchoring, shared \texttt{concept\_id}, cryptographic linkage, hash correctness, distinct \texttt{event\_id}, non-empty actor, timestamp monotonicity, payload schema conformance, action preconditions, hash format, canonical fields, and valid event type). The full conjunctive formalisation is in Appendix~E.

The status space is $S = \{\text{provisional},\; \text{validated},\; \text{deprecated},\; \text{forked},\; \text{archived}\}$. The status lattice is a join-semilattice with partial order:
\begin{equation}
    \text{provisional} < \text{forked} < \text{validated} < \text{deprecated} < \text{archived}
\end{equation}
For a chain $C$, let $C_{\text{life}}$ denote the lifecycle subsequence and let $f: \Sigma_{\text{life}} \to S$ map event types to their corresponding status values. The status derivation function $\delta(C)$ is the least upper bound (LUB) of all lifecycle events in the chain, computed as the maximum over the lattice order:
\begin{equation}
    \delta(C) = \begin{cases}
        \text{provisional} & \text{if } C_{\text{life}} = [] \\
        \max_{\prec}\{ f(e.\tau) \mid e \in C_{\text{life}} \} & \text{otherwise}
    \end{cases}
\end{equation}
where $\max_{\prec}$ is the maximum over the integer order of the lattice (1~provisional, 2~forked, 3~validated, 4~deprecated, 5~archived). This is computed in $O(|C_{\text{life}}|)$ time by a single scan of the lifecycle subsequence.

\textbf{Conflict resolution for $\delta$.} Under the CRDT LUB semantics, status is determined by the \emph{highest} lifecycle event in the lattice, not by the \emph{last} event. Once a chain reaches $\text{deprecated}$, no subsequent $\text{VALIDATE}$ or $\text{REGISTER}$ event can change the status back to $\text{validated}$ or $\text{provisional}$---the LUB of $\text{deprecated}$ with any lower status is always $\text{deprecated}$. Similarly, once a chain reaches $\text{archived}$, the status is permanently $\text{archived}$. This guarantees that status \emph{never regresses}, which is a fundamental property of monotonic CRDTs. Re-activation after deprecation requires a new $\text{REGISTER}$ event on a \emph{new} chain (creating a new genesis hash), not a reversal of the existing chain.

\subsubsection{Confidence Aggregation: Three Variants}

Let $V = [e \in C \mid e.\tau = \text{VALIDATE}]$ denote the validation subsequence. ADL Lite provides three confidence aggregation strategies.

\textbf{Variant 1: Default (G-Counter max).} The simplest strategy, used by default, takes the maximum confidence across all \texttt{VALIDATE} events (or \texttt{SNAPSHOT} if no validation has occurred):
\begin{equation}
    \gamma_{\text{default}}(C) = \begin{cases}
        0.0 & \text{if } V = [] \\
        \text{clamp}_{[0,1]}\bigl(\max\{ e.\text{payload}[\text{``confidence''}] \mid e \in V \}\bigr) & \text{otherwise}
    \end{cases}
\end{equation}
where $\max$ is the maximum over all validation events in the chain (G-Counter semantics), and $\text{clamp}_{[0,1]}(x) = \max(0, \min(1, x))$. Evaluated in $O(|V|)$ time by a single scan. Once a high-confidence validation is recorded, subsequent lower-confidence assertions cannot decrease the aggregate; this prevents malicious or erroneous validators from downgrading a concept after it has been validated.

\textbf{Variant 2: Bonus-formula aggregate ($\gamma_{\text{agg}}$).} This variant mitigates collusion by aggregating per-actor maxima and adding a bonus for cross-validation:
\begin{equation}
    \gamma_{\text{agg}}(C) = \begin{cases}
        0.0 & \text{if } V = [] \\
        \min\!\left(1.0,\; c_{\text{base}} + 0.05 \times (N_{\text{vals}} - 1)\right) & \text{otherwise}
    \end{cases}
\end{equation}
where $N_{\text{vals}} = |\text{actors}(V)|$ is the number of distinct validators, and:
\begin{equation}
    c_{\text{base}} = \max\!\left(0.5,\; \frac{1}{N_{\text{vals}}} \sum_{a \in \text{actors}(V)} \phi(a, V)\right)
\end{equation}
with $\phi(a, V) = \max\{e.\text{payload}[\text{``confidence''}] \mid e \in V \land e.\text{actor} = a\}$ being the maximum confidence reported by actor $a$. The $\min(1.0, \cdot)$ cap ensures boundedness; the $0.05$ bonus per distinct validator rewards cross-validation. Sensitivity analysis for different $(\beta, c_{\min})$ parameters appears in Appendix~E.

\textbf{Variant 3: Calibrated confidence ($\gamma_{\text{cal}}$).} This variant weights each validator's confidence by a per-actor accuracy score:
\begin{equation}
    \gamma_{\text{cal}}(C) = \begin{cases}
        0.0 & \text{if } V = [] \\
        \text{clamp}_{[0,1]}\!\left(\frac{\sum_{a \in \text{actors}(V)} \text{accuracy}_a \cdot \phi(a, V)}{\sum_{a \in \text{actors}(V)} \text{accuracy}_a}\right) & \text{otherwise}
    \end{cases}
\end{equation}
where $\text{accuracy}_a \in [0,1]$ is the accuracy score of actor $a$. This requires an external calibration profile (\texttt{.adl/calibration.yaml}).

\textbf{Disclaimer: calibration in the collaborative-audit model.} The $\text{accuracy}_a$ scores in $\gamma_{\text{cal}}$ are themselves self-reported in the Phase~1 collaborative-audit model (there is no external ground-truth source for validator accuracy). Consequently, $\gamma_{\text{cal}}$ provides a \emph{heuristic calibration} that redistributes confidence among validators, but it does \emph{not} prevent collusion: a colluding actor can report $\text{accuracy}_a = 1.0$ and drive the aggregate to any value in $[0,1]$. Full collusion resistance requires the authenticated identity and staking mechanisms planned for Phase~2/3 (FW4, FW9).

\textbf{Conflict resolution for $\gamma$.} All three variants resolve multiple \texttt{VALIDATE} events by computing per-actor maxima and then aggregating. For $\gamma_{\text{default}}$, the G-Counter ($\max$) over all \texttt{VALIDATE} events prevails. For $\gamma_{\text{agg}}$ and $\gamma_{\text{cal}}$, per-actor maxima are computed first, then aggregated. If two validators append \texttt{VALIDATE} events with different confidence values, each contributes its own maximum.\par
\textbf{Conflicting validators and time decay.} When validators disagree (e.g., \texttt{agent\_1} reports confidence 0.9 while \texttt{agent\_2} reports 0.5), the three variants resolve the conflict differently. $\gamma_{\text{default}}$ ignores the disagreement and takes the maximum value (0.9). $\gamma_{\text{agg}}$ preserves both values (via per-actor maxima) and rewards cross-validation with the bonus term. $\gamma_{\text{cal}}$ weights each validator by their accuracy score, so a high-accuracy validator's 0.9 dominates a low-accuracy validator's 0.5. Time decay is implemented via an optional EWMA (exponentially weighted moving average) layer with smoothing factor $\alpha = 0.3$: older validations receive geometrically decreasing weight, preventing stale confidence from dominating the aggregate. The EWMA layer is independent of the three variants and can be combined with any of them.

A fork operation $\text{Fork}(C, a)$ appends a $\textsc{Fork}$ event to the parent chain and creates a child chain with a fresh $\textsc{Register}$ event and a deterministic fork suffix appended to \texttt{concept\_id}. Child chains are independent: each child's derived state depends only on its own event sequence.

\textbf{Total order $\prec$ for conflict resolution.} Within a single chain, events are totally ordered by the cryptographic linkage (each event points to its predecessor via \texttt{previous\_event\_id}). For concurrent events across branches, we define a total order $\prec$ as:
\begin{equation}
    e_i \prec e_j \iff e_i.\text{timestamp} < e_j.\text{timestamp} \lor (e_i.\text{timestamp} = e_j.\text{timestamp} \land e_i.\text{event\_id} <_{\text{lex}} e_j.\text{event\_id})
\end{equation}
where $<_{\text{lex}}$ is lexicographic comparison of UUID strings. This order is total because timestamps are comparable and \texttt{event\_id} provides a tiebreaker. The append-only property of EventChains ensures that $\prec$ is consistent with the chain's internal order: if $e_i$ precedes $e_j$ in the chain, then $e_i.\text{timestamp} \leq e_j.\text{timestamp}$ (with equality possible only for simultaneous appends, resolved by \texttt{event\_id}).

\textbf{Concurrent conflict resolution.} When multiple agents append events to the same concept concurrently, the resulting events are ordered by $\prec$. For $\delta$: under the CRDT LUB semantics, the status is the maximum over all lifecycle events in the chain, independent of append order. A \textsc{Deprecate} event always dominates a \textsc{Validate} event (since $\text{deprecated} > \text{validated}$ in the lattice), and an \textsc{Archive} event dominates all other lifecycle events. For $\gamma$: the G-Counter ($\max$) over all \textsc{Validate} events determines the confidence, independent of append order. When a \textsc{Validate} and a \textsc{Deprecate} conflict, the \textsc{Deprecate} prevails in status but the confidence remains the maximum of all validations. Equivocation (the same actor appending contradictory events) is retained as auditable evidence, not resolved by the ordering.

\textbf{Theorem~1 (Determinism).} For any well-formed chain $C$, $\delta(C)$ is unique and is the least upper bound of all lifecycle events in $C$. \emph{Proof Sketch:} Let $C = [e_1, \ldots, e_n]$ be well-formed. By Well-Formedness Axiom~(3) (cryptographic linkage), $C$ is a totally ordered sequence. The lifecycle subsequence $C_{\text{life}} = \operatorname{filter}(C, \Sigma_{\text{life}})$ is therefore also totally ordered. If $C_{\text{life}} = []$, $\delta(C) = \text{provisional}$ by definition (unique). Otherwise, $C_{\text{life}}$ is a finite, non-empty sequence. The status map $f: \Sigma_{\text{life}} \to S$ assigns each lifecycle event a status value in the lattice. The LUB over a finite subset of a total order is the maximum element, which is unique because the lattice order is antisymmetric. Thus $\delta(C) = \max_{\prec}\{f(e.\tau) \mid e \in C_{\text{life}}\}$ is uniquely determined. The complexity claim follows because computing $C_{\text{life}}$ requires a single scan of $C$ ($O(|C|)$), and computing the maximum over $C_{\text{life}}$ is $O(|C_{\text{life}}|)$.

\textbf{Theorem~2 (Fork Determinism).} Let $C$ be a well-formed chain with $\delta(C) = s$ and let $C$ fork to $(C_{\text{fork}}, C')$. Then $\delta(C') = \text{provisional}$ and $\delta(C_{\text{fork}}) = \max(s, \text{forked})$ (the LUB of the parent's previous status and \textsc{Fork}). \emph{Proof Sketch:} By the fork definition, $C_{\text{fork}} = C \oplus [e_{\text{FORK}}]$ where $e_{\text{FORK}}.\tau = \text{FORK} \in \Sigma_{\text{life}}$. Under the CRDT LUB semantics, $\delta(C_{\text{fork}}) = \max_{\prec}\{f(e.\tau) \mid e \in (C_{\text{fork}})_{\text{life}}\} = \max(s, \text{forked})$. If the parent was previously \texttt{validated} ($s = \text{validated}$, with order 3), then $\max(3, 2) = 3$ and the parent remains \texttt{validated}; the \textsc{Fork} event is recorded as auditable evidence but does not downgrade the status. If the parent was \texttt{provisional} ($s = \text{provisional}$, with order 1), then $\max(1, 2) = 2$ and the parent becomes \texttt{forked}. The child chain $C' = [e'_{\text{REG}}]$ where $e'_{\text{REG}}.\tau = \text{REGISTER} \in \Sigma_{\text{life}}$ and $e'_{\text{REG}}.\text{previous\_event\_id} = \text{null}$. Since $C'$ contains exactly one event, $(C')_{\text{life}} = [e'_{\text{REG}}]$. By Theorem~1, $\delta(C') = f(\text{REGISTER}) = \text{provisional}$. The two derivations are independent because $C_{\text{fork}}$ and $C'$ are disjoint chains after the fork point: $C_{\text{fork}}$ appends to $C$ while $C'$ starts afresh. No subsequent event in either chain can affect the other's status derivation because status is derived per-chain (Theorem~1).

\textbf{Theorem~3 (Transition Monotonicity).} For any well-formed $C$ and event $e_{\text{new}}$, the transition from $s = \delta(C)$ to $s' = \delta(C \oplus [e_{\text{new}}])$ is valid iff $e_{\text{new}}.\tau \in \Sigma_{\text{life}}$. \emph{Proof Sketch:} By structural induction on $|C|$. \textbf{Base case:} $C = [e_{\text{REG}}]$ with $\delta(C) = \text{provisional}$. (The empty chain is not well-formed; every well-formed chain has a genesis \textsc{Register} event as its first element.) For any $e_{\text{new}}$: if $e_{\text{new}}.\tau \notin \Sigma_{\text{life}}$, then $(C \oplus [e_{\text{new}}])_{\text{life}} = [e_{\text{REG}}]$, so $s' = \text{provisional} = s$ (no transition). If $e_{\text{new}}.\tau \in \Sigma_{\text{life}}$, then $s' = f(e_{\text{new}}.\tau)$, which the ActionExecutor precondition system permits only if $(\text{provisional}, f(e_{\text{new}}.\tau))$ is a valid edge in the status transition machine. \textbf{Inductive step:} Assume the theorem holds for all chains of length $n$. Let $C$ have length $n+1$ with $\delta(C) = s$. Consider $e_{\text{new}}$: if $e_{\text{new}}.\tau \notin \Sigma_{\text{life}}$, then $(C \oplus [e_{\text{new}}])_{\text{life}} = C_{\text{life}}$, so $s' = s$ (no transition). If $e_{\text{new}}.\tau \in \Sigma_{\text{life}}$, then $(C \oplus [e_{\text{new}}])_{\text{life}} = C_{\text{life}} \oplus [e_{\text{new}}]$, so $s' = f(e_{\text{new}}.\tau)$. By the ActionExecutor precondition system, $e_{\text{new}}$ can be appended only when the precondition rules are satisfied against the current snapshot derived from $C$. These preconditions encode exactly the valid transitions of the status machine, so $(s, s')$ must be a valid edge. Thus the theorem holds for chains of length $n+1$. By induction, it holds for all well-formed chains.

\textbf{Theorem~4 (Confidence Boundedness).} For any well-formed $C$, $\gamma_{\text{default}}(C) \in [0, 1]$, $\gamma_{\text{agg}}(C) \in [0, 1]$, and $\gamma_{\text{cal}}(C) \in [0, 1]$. \emph{Proof Sketch:} For $\gamma_{\text{default}}$: if $V = []$, $\gamma_{\text{default}}(C) = 0.0 \in [0,1]$. Otherwise, the payload \texttt{confidence} field is clamped to $[0,1]$ by Pydantic validation at event construction time, so $\gamma_{\text{default}}(C) = \text{clamp}_{[0,1]}(c) \in [0,1]$. For $\gamma_{\text{agg}}$: if $V = []$, $\gamma_{\text{agg}}(C) = 0.0$. Otherwise, each $\textsc{Validate}$ event carries confidence $\in [0,1]$; the per-actor maxima $\phi(a, V)$ are therefore in $[0,1]$, and their mean is a convex combination in $[0,1]$. The $\max(0.5, \cdot)$ floor ensures $c_{\text{base}} \geq 0.5$, and the $\min(1.0, \cdot)$ cap ensures $\gamma_{\text{agg}}(C) \leq 1.0$. For $\gamma_{\text{cal}}$: the weighted sum of values in $[0,1]$ with non-negative weights is a convex combination in $[0,1]$, and the outer clamp preserves this bound. Thus all three variants are bounded in $[0,1]$.

\textbf{Theorem~5 (Confidence Monotonicity).} For any well-formed $C$ and any $\textsc{Validate}$ event $e_{\text{new}}$ with payload confidence $c'$, $\gamma_{\text{default}}(C \oplus [e_{\text{new}}]) \geq \gamma_{\text{default}}(C)$. \emph{Proof Sketch:} $\gamma_{\text{default}}$ takes the maximum confidence over all $\textsc{Validate}$ events in $V$ (G-Counter semantics). Appending $e_{\text{new}}$ with $e_{\text{new}}.\tau = \textsc{Validate}$ adds $c'$ to the set of validation confidences. Therefore $\gamma_{\text{default}}(C \oplus [e_{\text{new}}]) = \max(\gamma_{\text{default}}(C), \text{clamp}_{[0,1]}(c')) \geq \gamma_{\text{default}}(C)$. Strict inequality holds if $c' > \gamma_{\text{default}}(C)$ and $c' \in [0,1]$. For $\gamma_{\text{agg}}$: monotonicity holds because appending a new $\textsc{Validate}$ from a distinct validator increases $N_{\text{vals}}$ by 1 (adding $0.05$ to the bonus) and $c_{\text{base}}$ is a weighted mean that includes the new confidence. For duplicate validators, $\phi'(a, V') = \max(\phi(a, V), c') \geq \phi(a, V)$, so $c'_{\text{base}} \geq c_{\text{base}}$. The bonus term is unchanged or increased, so $\gamma_{\text{agg}}(C') \geq \gamma_{\text{agg}}(C)$. (The optional $\gamma_{\text{cal}}$ aggregate does not guarantee monotonicity unless all validators' accuracy scores are non-decreasing.) Full details are in Appendix~E.

\textbf{Theorem~6 (Status--Confidence Consistency).} If $\delta(C) = \text{validated}$, then $\gamma_{\text{default}}(C) \geq 0.5$, $\gamma_{\text{agg}}(C) \geq 0.5$, and $\gamma_{\text{cal}}(C) \geq 0.5$ (provided all validators have accuracy $\geq 0.5$). \emph{Proof Sketch:} $\delta(C) = \text{validated}$ implies that the LUB of all lifecycle events in $C$ is $\text{validated}$, which requires at least one $\textsc{Validate}$ event in $C_{\text{life}}$ (since no other lifecycle event type maps to $\text{validated}$). Thus $V \neq \emptyset$. Every \texttt{validate} action precondition requires payload confidence $\geq 0.5$ (enforced by the ActionExecutor via \texttt{adl\_core\_ontology.yaml}). For $\gamma_{\text{default}}$: the G-Counter max over all $\textsc{Validate}$ events includes at least one event with confidence $\geq 0.5$, so $\gamma_{\text{default}}(C) \geq 0.5$. For $\gamma_{\text{agg}}$: every $\phi(a, V) \geq 0.5$, so the mean $c_{\text{base}}^{\text{raw}} \geq 0.5$, and $c_{\text{base}} = \max(0.5, c_{\text{base}}^{\text{raw}}) \geq 0.5$. The bonus term is non-negative, so $\gamma_{\text{agg}}(C) \geq 0.5$. For $\gamma_{\text{cal}}$: if all $\text{accuracy}_a \geq 0.5$, the weighted mean of values $\geq 0.5$ is itself $\geq 0.5$.

\textbf{Theorem~7 (Well-Formedness Preservation).} For any well-formed chain $C$ and any event $e_{\text{new}}$ satisfying the preconditions of its action type, the extended chain $C' = C \oplus [e_{\text{new}}]$ is well-formed. \emph{Proof Sketch:} Each $\text{WF}_i$ axiom is verified for $C'$: genesis remains unchanged, \texttt{concept\_id} shared, linkage preserved, hash computed, and fresh \texttt{event\_id} ensures distinctness. Full details are in Appendix~E.

We provide proof sketches for all seven core theorems (Theorems 1--7) through rigorous natural-language arguments over the closed alphabet $\Sigma$; full details are in Appendix~E. Machine-checked proofs for unbounded chains are planned for future work (FW10, FW12).

\subsubsection{Formal Function Signatures and a Worked Example}

\textbf{Explicit function signatures.} The derivation functions are defined as:
\begin{align}
    \delta &: \text{EventChain} \to S \\
    \delta(C) &= \begin{cases}
        \text{provisional} & \text{if } C_{\text{life}} = \emptyset \\
        f\bigl(\tau_{\text{last}}(C_{\text{life}})\bigr) & \text{otherwise}
    \end{cases} \nonumber \\[1em]
    \gamma &: \text{EventChain} \to [0,1] \\
    \gamma(C) &= \begin{cases}
        0.0 & \text{if } V = \emptyset \\
        \text{clamp}_{[0,1]}\bigl(e_{\text{last}}(V).\text{payload}[\text{``confidence''}]\bigr) & \text{otherwise}
    \end{cases} \nonumber
\end{align}
where $S$ is the status space, $\text{EventChain}$ is the set of well-formed chains, $C_{\text{life}}$ is the lifecycle subsequence, $V$ is the validation subsequence, and $f : \Sigma_{\text{life}} \to S$ is the total event-type-to-status mapping.

\textbf{Small-step operational semantics.} Appending an event $e_{\text{new}}$ to a chain $C$ is denoted $C \oplus [e_{\text{new}}]$. The transition is a pure function:
\begin{equation}
    \langle C \rangle \xrightarrow{e_{\text{new}}} \langle C \oplus [e_{\text{new}}] \rangle
    \quad \text{where} \quad
    \delta(C \oplus [e_{\text{new}}]) = \begin{cases}
        \delta(C) & \text{if } e_{\text{new}}.\tau \notin \Sigma_{\text{life}} \\
        f(e_{\text{new}}.\tau) & \text{otherwise}
    \end{cases}
\end{equation}
This is deterministic: the new state depends only on the old state and the appended event, with no side effects or stored mutable fields.

\textbf{Worked example.} Consider a 5-event chain for the concept \texttt{disc-capital-trap}. After the genesis \textsc{Register} event (Step~1), a \textsc{Relate} event (Step~2, communication) leaves status unchanged at \textit{provisional}. A \textsc{Validate} event (Step~3) with confidence 0.70 transitions status to \textit{validated} and sets $\gamma = 0.70$. An \textsc{Evidence} event (Step~4) records audit data without changing derived state. A second \textsc{Validate} (Step~5) with confidence 0.85 updates $\gamma$ to 0.85, demonstrating Theorem~5 (Monotonicity). If agent\_2 disagrees at Step~5, they may append a \textsc{Fork} event: the parent becomes \textit{forked} and the child begins at \textit{provisional}, both independently well-formed.

\subsection{Comparison with Formal Event Frameworks}\label{subsec:event-frameworks}

ADL Lite's event model may be compared to Event Calculus, Situation Calculus, and Description Logic. In Event Calculus terms, $\delta(C)$ computes the fluent $\text{Status}(c, t)$ by scanning the ordered event sequence, equivalent to EC's interval-based reasoning but computationally simpler ($O(n)$ vs.~$O(n^2)$). ADL Lite avoids the frame problem by construction: there are no stored mutable fields, so all properties are recomputed from the event sequence. The derivation functions $\delta$ and $\gamma$ are computable in linear time ($\mathcal{O}(n)$ and $\mathcal{O}(1)$, respectively). They correspond to DL-Lite$_{\mathcal{R}}$ path queries and do not require full OWL-DL expressivity (NExpTime-complete). A full discussion is in Appendix~H.

\subsection{Trust Model and Security Boundaries}\label{subsec:trust-model}

The cryptographic hash chain ensures content integrity: any modification breaks the chain linkage and is detected by $\text{VerifyIntegrity}(C)$. The scope of the current collaborative-audit model is deliberately limited: hash chains detect tampering but do not prevent it, and they provide no actor authentication.

\subsubsection{Trust Assumptions}

We assume (1)~trusted storage (Git or filesystem preserving history integrity, with signed commits providing tamper evidence), (2)~trusted authoring environment (compromised agents inject events, which are recorded as tamper-evident evidence), and (3)~clock skew tolerance (ordering uses cryptographic linkage, not timestamps). Actor identity in the current implementation is a self-declared string; impersonation and replay are discussed below. Future authentication details are in Appendix~K.

\subsubsection{Current Trust Boundary: What Is and Is Not Detectable}

The hash chain detects \emph{content} tampering (modification of existing events) because any change breaks the hash linkage. However, the current model cannot detect the following \emph{structural} and \emph{identity} attacks: (i)~\emph{Actor impersonation:} any agent may declare an arbitrary string identifier; there is no cryptographic binding to a real-world identity. (ii)~\emph{Replay:} an event (or an entire chain) can be copied and re-submitted under a different concept identifier. (iii)~\emph{Sybil creation:} a single agent can create multiple pseudonymous identities and self-validate using each. (iv)~\emph{Selective omission:} an agent may withhold evidence that would contradict their claim; the chain provides no mechanism to prove that a missing event \emph{should} have been present. (v)~\emph{Historical rewrite:} an actor with repository-level access (e.g., Git push access) can force-push a rebased history, replacing the chain with a new genesis. These attacks are detectable by external audit (comparing a local clone against a signed reference), but they are not \emph{prevented} by the chain itself. This characterisation explains why we describe the current model as a \emph{collaborative non-Byzantine} setting.

\subsubsection{Collusion Vulnerability Analysis}\label{subsubsec:collusion-analysis}

The confidence aggregation function $\gamma(C)$ is vulnerable to collusion in the current collaborative-audit model because it lacks authentication and incentive compatibility.

\begin{lemma}[Collusion Vulnerability]\label{lem:collusion-vulnerability}
In the current collaborative-audit model, $\gamma(C)$ returns the confidence of the most recent \texttt{VALIDATE} event (clamped to $[0,1]$). A single actor can therefore drive $\gamma(C)$ to any value in $[0,1]$ by appending a \texttt{VALIDATE} event with the desired confidence, achieving maximum confidence without independent validation.
\end{lemma}
\begin{proof}
By definition, $\gamma(C)$ scans the chain for the last \texttt{VALIDATE} event and returns the clamped confidence value: $\text{clamp}_{[0,1]}(e_{\text{last}}.\text{payload}[\text{``confidence''}])$. A colluding actor appending an event with $\text{confidence} = 1.0$ yields $\gamma(C) = 1.0$.
\end{proof}

\begin{lemma}[Minimum Validator Threshold Mitigation]\label{lem:threshold-mitigation}
Let $N_{\min} \geq 2$ be a minimum validator threshold enforced as a precondition for \texttt{VALIDATE}. Then a single colluding actor cannot trigger the $\text{validated}$ status; the collusion cost rises to $k \geq N_{\min}$.
\end{lemma}
\begin{proof}
The precondition $\langle N_{\text{vals}}, \text{GTE}, N_{\min} \rangle$ is evaluated before appending; for $N_{\text{vals}} = 1 < N_{\min}$, the event is rejected.
\end{proof}

\textbf{Implications.} Lemma~\ref{lem:collusion-vulnerability} proves that the current model's confidence aggregation lacks collusion resistance because a single actor controls the most recent validation. Lemma~\ref{lem:threshold-mitigation} shows that a minimum-validator threshold provides partial mitigation. Full collusion resistance requires EWMA calibration and per-band correction (implemented as described in \S\ref{subsec:calibration}) and a staking-based mechanism (FW9).

\subsubsection{Known Limitations and Planned Mitigations}

The current model has four limitations: (1) no actor identity verification for legacy string identifiers (we have implemented a minimal \texttt{did:key} layer as described below; full W3C LD-Proofs and DIDs remain future work); (2) no Byzantine resilience; (3) no equivocation prevention (contradictory events are recorded as auditable evidence); (4) genesis immutability relies solely on hash-chain detection and signed Git commits.

\textbf{Identity layer gap and DIDs/VCs survey.} The current self-declared string identifier model lacks the cryptographic binding of decentralized identity standards. A survey of DIDs and VCs by Mazzocca et al.~\cite{mazzocca2024didvc} underscores the importance of self-sovereign identity systems for agent ecosystems: DIDs provide persistent, cryptographically verifiable identifiers without centralized registries, while VCs enable selective disclosure of capability attestations. ADL Lite's Phase~2 roadmap directly addresses this gap by integrating \texttt{did:key} resolution (Phase~1.5) and W3C Linked Data Proofs (Phase~2), enabling per-event signatures and key rotation that would satisfy the identity requirements identified in the survey.\par
\textbf{Three-phase authentication roadmap.} \textbf{Phase 1 (current):} collaborative-audit model with self-declared string identifiers and SHA-256 hash-chain integrity. \textbf{Phase 1.5 (implemented, pre-release):} minimal \texttt{did:key} resolution without external dependencies; Ed25519 signatures in the \texttt{signature} field; \texttt{verify\_signature()} validates both DID-derived and registry-derived keys. This demonstrates cryptographic identity beyond self-declared strings but remains scoped to the collaborative-audit model. \textbf{Phase 2 (planned):} full Ed25519 key registry (\texttt{key\_registry.py}) with \texttt{KEY\_ROTATE} and \texttt{KEY\_REVOKE} events; DID document storage with \texttt{did:web} and \texttt{did:ethr} support; W3C Linked Data Proofs. \textbf{Phase 3 (future):} BFT transport layer; CRDT-style merge with semantic policies; Ontological Assertion Market (staking/slashing for incentive-compatible validation). Authenticated identities would change the threat model: Sybil attacks (L3a) become impossible, and collusion cost rises to economic stakes (FW9).

\textbf{Phase 1.5} provides Ed25519 \texttt{did:key} resolution and signature verification (\S\ref{subsec:trust-model}), which mitigates impersonation, replay, and genesis replacement relative to string identifiers, but does not prevent Sybil creation (a single actor can still generate multiple \texttt{did:key}s) or collusion (no economic cost).

The CRDT merge semantics described here are now \textbf{implemented in the production codebase}. As of v0.3.5, \texttt{EventChain.status} and \texttt{EventChain.confidence} derive via CRDT lattice operations rather than the earlier last-event rule. The status lattice is a join-semilattice with partial order $\text{provisional} < \text{forked} < \text{validated} < \text{deprecated} < \text{archived}$; the status of a chain is the least upper bound (LUB) of all lifecycle events in the chain, computed as $\max$ over the integer order. The confidence is a G-Counter (max) over all \texttt{VALIDATE} events, so once a high-confidence validation is recorded, subsequent lower-confidence assertions cannot decrease the aggregate. This prevents malicious or erroneous validators from downgrading a concept after it has been validated. Theorem~9 below is now a \emph{verified property} of the current system, not a future capability.

The CRDT merge operates at the \emph{set} level (events as elements), while the hash chain operates at the \emph{sequence} level (events as ordered nodes). When two branches $B_1$ and $B_2$ are merged, the resulting event set $B_1 \cup B_2$ is re-linearised by sorting on \texttt{timestamp} (with \texttt{event\_id} tie-breaking). The hash chain is then recomputed over this merged sequence: each event's \texttt{previous\_event\_id} and \texttt{prev\_hash} are updated to point to its predecessor in the new ordering. This means the merged chain has a \emph{different} hash sequence from either parent branch, but all original event content is preserved (via its immutable \texttt{event\_id} and payload). The hash chain therefore provides \emph{post-merge integrity}, not \emph{pre-merge ancestry preservation}. For applications requiring ancestry proofs, the original branch hashes can be stored as metadata in a \textsc{MERGE} event payload.

\textbf{Handling conflicting lifecycle events.} Under the CRDT LUB semantics, conflicting lifecycle events are resolved by the lattice order rather than by timestamp. If a chain contains both a \texttt{VALIDATE} and a \texttt{DEPRECATE} event, the LUB is $\text{deprecated}$ (since $\text{deprecated} > \text{validated}$ in the lattice). Similarly, once a chain reaches $\text{archived}$, no subsequent lifecycle event can change the status---the LUB of $\text{archived}$ with any other status is always $\text{archived}$. This is semantically stronger than the earlier last-event rule: it guarantees that \emph{status never regresses}, which is a fundamental property of monotonic CRDTs. The calibration layer ($\gamma_{\text{cal}}$) still weights validators by accuracy, but the LUB guarantees that high-accuracy validations are never erased by low-accuracy deprecations (the confidence G-Counter preserves the maximum). Equivocation---both events from the same actor---is retained as explicit auditable evidence of disagreement and is not resolved by the merge itself.

\textbf{Theorem~9 (CRDT Convergence).} For any well-formed concurrent branches $B_1, B_2$ derived from the same genesis event, the LWW-Set merge satisfies commutativity, associativity, and idempotence, and $\delta(\text{Merge}(B_1, B_2))$ is uniquely determined. \emph{Proof Sketch:} (i)~\emph{Commutativity:} $B_1 \cup B_2 = B_2 \cup B_1$ by set union symmetry. (ii)~\emph{Associativity:} $(B_1 \cup B_2) \cup B_3 = B_1 \cup (B_2 \cup B_3)$ by set union associativity. (iii)~\emph{Idempotence:} $B \cup B = B$ by set idempotence; deduplication by \texttt{event\_id} ensures no duplicates. (iv)~\emph{Determinism of $\delta$:} the merged set contains all lifecycle events from both branches; the LUB over the status lattice is uniquely determined, so $\delta$ is independent of merge order. We verify Theorem~9 through executable assertions (722 tests pass). Machine-checked proofs for unbounded chains are planned for future work (FW10, FW12).

\textbf{Migration from last-event to CRDT: completed.} The earlier last-event fork resolution was a special case of CRDT merge where the branch set size is 1 (no concurrent branches). The migration to full CRDT compliance has been completed in three phases. \emph{Phase~1 (completed):} last-event with total order $\prec$; status and confidence derived from the most recent event. \emph{Phase~2 (completed):} CRDT lattice semantics for status (LUB) and confidence (G-Counter max); all events commute because they are append-only and immutable. \emph{Phase~3 (future work):} semantic merge policies for multi-way concurrent merges with conflicting lifecycle events. Under Phase~2, \texttt{REGISTER} events commute (they create independent genesis events); \texttt{RELATE}, \texttt{EVIDENCE}, and \texttt{SEAL} events commute (they are communication events that do not affect $\delta$); \texttt{VALIDATE} events from distinct actors commute (each contributes to the per-actor maxima in $\gamma_{\text{agg}}$ and $\gamma_{\text{cal}}$). Conflicting lifecycle events---\texttt{VALIDATE} vs. \texttt{DEPRECATE}, or multiple \texttt{VALIDATE} events from the same actor (equivocation)---require conflict-resolution policies for Phase~3: (a)~quorum-based resolution ($\geq N_{\min}$ validators for \texttt{VALIDATE} to override \texttt{DEPRECATE}); (b)~accuracy-weighted ordering (higher-accuracy actors' events prevail); (c)~stake-weighted resolution (higher-stake validators' events prevail, planned for FW9). Blocklace~\cite{blocklace2024} provides the BFT transport layer for Phase~3: its cryptographic hash DAG enables equivocation detection and Byzantine exclusion, while ADL Lite provides the application semantics (lifecycle derivation, precondition enforcement). The integration architecture is: Blocklace DAG as the transport layer; ADL Lite EventChain as the application layer; a shim layer maps Blocklace events to ADL Lite events and resolves conflicts using the semantic policies above.

\subsection{Calibration Layer}\label{subsec:calibration}

The calibration layer mitigates overconfidence in the default $\gamma(C)$ aggregation by offering four complementary strategies, all implemented in \texttt{calibration.py}: \textbf{linear calibration} (per-actor accuracy-weighted confidence), \textbf{EWMA time-decay} (exponentially weighted moving average with smoothing factor $\alpha = 0.3$), \textbf{epistemic context} (domain-specific accuracy scores per actor per domain), and \textbf{per-band correction} (low-confidence values $[0.0, 0.3)$ receive $+0.15$ boost; high-confidence $[0.7, 1.0]$ receive $-0.10$ dampening). These strategies operate independently and can be combined. The default EventChain uses CRDT G-Counter semantics for confidence ($\max$ over all \texttt{VALIDATE} events) and join-semilattice LUB for status; we invoke the calibration layer explicitly when cross-validation depth or collusion risk warrants more conservative aggregation.

\subsection{Semantic Web Interoperability}\label{subsec:semantic-web-interop}

ADL Lite implements two Semantic Web export pathways. \textbf{Bidirectional OWL~2 DL import/export} (\texttt{owl\_export.py} / \texttt{owl\_import.py}) converts \texttt{ADLDocument} to OWL~2 DL in Turtle or RDF/XML, mapping concepts to \texttt{NamedIndividual}s, status to annotation properties, and events to typed individuals. Round-trip validation preserves concept identity, status, confidence, and event count. \textbf{RDF-star / SPARQL-star} (\texttt{rdfstar\_export.py}) converts events to embedded triples, enabling provenance queries such as ``find all concepts validated by agent X with confidence $>0.8$ in the last 30 days.'' Three additional pathways remain future work (FW8): SHACL validation, PROV-O mapping, and JSON-LD serialization.
