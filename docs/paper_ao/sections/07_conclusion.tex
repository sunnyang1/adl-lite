\subsection{Summary of Contributions}\label{subsec:summary}

This paper presented ADL Lite, an event-first capability-lifecycle registry, and evaluated it through ontological analysis, formal semantics, and empirical validation.

\textbf{Ontological analysis.} We analysed ADL Lite's core categories through BFO, DOLCE, and UFO, formalizing identity conditions, relation lifecycle semantics, and three forms of ontological dependence. We resolved the apparent tension between the BFO constraint that generically dependent continuants cannot depend on occurrents and the causal origin of concepts by distinguishing historical causation from ontological dependence.

\textbf{Formal derivation semantics.} We provided a complete formal foundation---event alphabet, well-formedness predicate, three confidence-aggregation variants ($\gamma_{\text{default}}$, $\gamma_{\text{agg}}$, $\gamma_{\text{cal}}$), fork determinism with a total order $\prec$ for conflict resolution, and seven core proven properties (Theorems 1--7). We further analysed expressive power in relation to Event Calculus and Description Logic.

\textbf{Empirical validation.} We verified the architecture of the registry on 201 EventChains (9,300 events). The system achieved zero integrity failures, 100\% derivation accuracy, perfect precondition enforcement (P = R = F1 = 1.0), and linear scalability ($20{,}847$ events per second).

ADL Lite is not a competing ontology production method; it is a complementary governance layer that records, validates, and governs the lifecycle of capabilities produced by any agent pipeline.\par

\subsection{Future Work}\label{subsec:future-work}

Future work is organized into four thematic clusters that address the limitations identified in Section~\ref{subsec:limitations}. Each item represents a capability that has not yet been implemented in the current codebase.

\textbf{Priority ordering and three-phase roadmap.} The most urgent items are \textbf{FW1} (automated OWL~2 DL alignment and ROBOT validation), \textbf{FW4} (decentralized identity and cryptographic authentication), and \textbf{FW9} (game-theoretic incentive design). Medium-term items include \textbf{FW5} (domain evaluation with human experts), \textbf{FW6} (SHACL-governed document validation), and \textbf{FW10/12} (machine-checked proofs and mechanized ontological axioms). Long-term directions are \textbf{FW7} (LLM-guided concept discovery) and \textbf{FW8/13} (Semantic Web interoperability and agent standards integration).

The roadmap is organized into three phases that align with the authentication progression described in \S\ref{subsec:trust-model}: \textbf{Phase 1} (current, v0.2--v0.4): collaborative-audit model with self-declared string identifiers, SHA-256 hash-chain integrity, and deterministic last-event fork resolution. \textbf{Phase 1.5} (implemented, pre-release): minimal \texttt{did:key} resolution and Ed25519 signature verification, demonstrating cryptographic identity beyond self-declared strings. \textbf{Phase 2} (planned, v0.5+): full DID registry with key rotation/revocation, W3C LD-Proofs, and Merkle-tree batch verification, replacing string identifiers with authenticated DIDs. \textbf{Phase 3} (future, v1.0+): BFT transport layer, CRDT-style merge with semantic policies, and Ontological Assertion Market (staking/slashing), enabling open-ecosystem deployment with adversarial robustness.

\textbf{Migration path and hash stability.} The SHA-256 hash of each event is computed from the canonical serialization (\texttt{Canon($e$)}), which includes the \texttt{event\_id}, \texttt{timestamp}, \texttt{actor}, \texttt{payload}, and \texttt{previous\_event\_id}. Adding stronger authentication does \emph{not} change existing hashes because the signature is stored \emph{outside} the canonical payload (in a separate \texttt{signature} envelope) or appended to the event after hash computation. This ensures backward compatibility: historical events remain valid under their original hashes; new verifiers can optionally check the signature envelope while legacy verifiers continue to check the hash alone. The signature envelope includes a \texttt{proof\_type} field (e.g., \texttt{Ed25519Signature2020}, \texttt{DataIntegrityProof}) that specifies the signature scheme, enabling future migration to post-quantum signatures without invalidating existing events. The migration path is: Phase 1.5 adds Git commit signatures (GPG or SSH) at the repository level, preserving all event hashes; Phase 2 adds per-event Ed25519 signatures as wrappers around existing hashes; Phase 3 replaces self-declared \texttt{actor} strings with DIDs (\texttt{did:key} or \texttt{did:web}) and LD-Proofs, with the hash remaining the primary identifier and the signature providing non-repudiation.

\textbf{Impact on $\delta$, $\gamma$, and preconditions.} The authentication upgrade does not alter the $\delta$ or $\gamma$ functions because the signature field is stored outside the canonical payload (it is not included in the hash computation). The only change is in the ActionExecutor's precondition evaluation: Phase 2 adds a new precondition rule (\texttt{signature EXISTS} and \texttt{signature\_valid EQ true}) that is evaluated before the standard action preconditions. Phase 3 adds two additional preconditions: (i) \texttt{actor} must be a valid DID (format check via regex), and (ii) the LD-Proof must verify against the DID document (cryptographic verification). The $\delta$ and $\gamma$ functions are unaffected by these additional preconditions because they are evaluated after the event is appended (the preconditions only control whether the event is accepted). The calibration layer ($\gamma_{\text{cal}}$) benefits from Phase 3 because the \texttt{accuracy}$_a$ scores can be bound to the DID (via the DID document's service endpoints or reputation credentials), making the calibration layer more robust against Sybil attacks.

\subsubsection{Formal Foundations and Ontological Alignment}
FW1: Automated OWL~2 DL alignment of ADL Lite categories with BFO, DOLCE, and UFO using ontology alignment tools (e.g., LogMap, AML), followed by ROBOT consistency validation. While bidirectional Turtle and RDF/XML import/export are already implemented (\S\ref{subsec:semantic-web-interop}), the automated alignment and machine-checkable axiomatization remain future work. FW10: Machine-checked proofs of the seven core theorems in TLA$^+$ or Coq (Iris separation logic), building on Nieto's CRDT formalization~\cite{nieto2024crdt}. FW12: Mechanized verification of the two-level ontological axioms and precondition decidability.

\subsubsection{Authentication, Security, and Scale}
FW4: Decentralized identity via W3C DIDs and Linked Data Proofs, with Merkle-tree batch verification~\cite{linked_data_proofs,crosby_wallach} and Blocklace-inspired equivocation detection~\cite{blocklace2024}. This includes W3C Verifiable Credentials with selective disclosure (BBS+ signatures) for privacy-preserving actor attestation. While Ed25519 key registration and Git binding are operational, full DID resolution and Merkle/Verkle batch verification remain unimplemented.

\subsubsection{Incentive Design and Domain Evaluation}
FW9: \textit{Ontological Assertion Market}---a staking-based mechanism where validators deposit collateral to validate concepts, and slashing penalizes overconfidence or collusion. This creates game-theoretic guarantees for truthful reporting that the current calibration layer (\S\ref{subsec:calibration}) cannot provide. FW5: Domain effectiveness evaluation on the IBM AML HI-Small dataset with IRB-approved human expert ratings, moving beyond synthetic pattern injection. FW15: \textit{Planned domain-level evaluation protocol}---a within-subjects study comparing ADL Lite lifecycle governance, PROV-O with SHACL validation, and nanopublications with Trusty URIs on delegation accuracy, trust calibration, and audit completeness, with 30 participants (10 experts, 10 LLM agents, 10 novices) across 20 suspicious transaction patterns. FW6: SHACL-governed L2 Markdown validation for [Observation], [Reasoning], [Conclusion] sections, enabling machine-checkable document structure independent of the Python runtime. While Pydantic-based template validation is operational (E20), SHACL provides a standards-compliant pathway for non-Python environments. FW7: LLM-guided concept discovery with RAG-injected domain knowledge~\cite{llm4acoe} and human-in-the-loop review, replacing the brittle \texttt{\_id} suffix matching in \texttt{DataImporter.discover\_classes()}.

\subsubsection{Semantic Web Interoperability and Agent Standards}
FW8: Three Semantic Web export pathways: (i)~\textbf{SHACL validation:} shape constraints for L1 schema conformance and L2 template structure; (ii)~\textbf{PROV-O provenance:} lifecycle events mapped to \texttt{prov:Activity}, actors to \texttt{prov:Agent}, and chains to \texttt{prov:Entity} with \texttt{wasGeneratedBy} links; (iii)~\textbf{JSON-LD serialization:} canonical event JSON with \texttt{@context} mapping to the ADL Lite ontology namespace. While OWL 2 DL bidirectional import/export and RDF-star/SPARQL-star are already implemented (\S\ref{subsec:semantic-web-interop}), SHACL and JSON-LD remain future work (the PROV-O mapping is already implemented; \S\ref{subsec:prov-mapping}). FW13: Integration with emerging agent registry standards (MCP, A2A, AGNTCY, NANDA, Entra) through adapter modules that map ADL Lite EventChains to their respective capability and identity formats.
