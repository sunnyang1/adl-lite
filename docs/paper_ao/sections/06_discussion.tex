\subsection{Capability-First Governance: Ontological Implications}\label{subsec:ontological-implications}

The empirical results support the event-first design, but the deeper contribution lies in four ontological questions that emerge from the analysis.

\textbf{Q1: What is the ontological status of a capability?} In ADL Lite, a capability is a generically dependent continuant that exists only from its genesis event (\texttt{REGISTER}) onward; prior to that, there is only potential. This parallels BFO, where a process exists only from the moment it begins.

\textbf{Q2: Are status and confidence real or derived properties?} Status is computed by $\delta(C)$ and has no existence independent of the chain, aligning with DOLCE's view that some properties are constructed by cognitive agents rather than discovered~\cite{dolce}.

\textbf{Q3: What happens to a capability when it is deprecated?} Deprecation appends a \texttt{DEPRECATE} event; the capability retains its \texttt{concept\_id} but its status becomes \texttt{deprecated}, preserving a complete audit trail. The entity itself is not destroyed---its identity persists---but its governance status changes, reflecting the distinction between identity and status that is central to the event-first design.

\textbf{Q4: What is the ontological cost of the event-first stance?} The event-first model makes historical queries natural (e.g., ``what was the status at time $t$?''), but certain snapshot queries require scanning the entire chain. For example, computing the set of all currently \texttt{validated} concepts requires evaluating $\delta(C)$ for every chain, which is $O(n)$ per chain. This is a deliberate design choice: event-first trades query-time computation for the elimination of stored mutable state and the consistency defects it introduces. The complexity is linear and bounded (Theorem~1), so the cost is predictable and acceptable for the target deployment scale ($10^5$--$10^6$ events).

\subsection{Capability Registry Contribution}\label{subsec:conceptual-modeling}

ADL Lite contributes to capability registry practice in three respects, each addressing a distinct practical need.\par
\textbf{Layered capability documentation.} The L1--L4 document model spans identity, narrative, assertion, and action, serving as a documentation methodology applicable to domains beyond multi-agent governance.

\textbf{Event-driven capability lifecycle modeling.} ADL Lite represents lifecycle states as event histories rather than as static attributes or state machines---an event-driven perspective particularly valuable for domains with high capability velocity.\par
\textbf{Collaborative capability governance.} The precondition system and closed action registry provide a lightweight, declarative, and human-readable mechanism for governing capability evolution, thereby lowering the barrier to collaborative authoring while maintaining machine-checkable constraints.

\subsection{Limitations}\label{subsec:limitations}

The results must be interpreted in light of twelve limitations, each of which is addressed by a corresponding Future Work item in Section~\ref{subsec:future-work}.\par
\textbf{L1. Informal foundational alignment.} The ontological analysis is conceptual and narrative in nature, relying on expert judgment and manual cross-referencing with BFO, DOLCE, and UFO literature. It is not formalized as OWL~2~DL axioms nor validated with automated alignment tools (e.g., LogMap, AML). (See~\S\ref{subsec:future-work}~FW1.)\par
\textbf{L2. Append-only accumulation and scale limits.} The append-only design inevitably leads to junk accumulation, redundancy, and preference drift. While feasibility at $10^5$--$10^6$ events was originally projected from linear scaling arguments, empirical validation at that scale has now been performed (E19, Section~\ref{subsec:comparative-analysis}): ADL Lite ingests $10^6$ concepts (2$\times$$10^6$ events) in 87.0~s at 22,987 events per second, confirming the linear projection. (See~\S\ref{subsec:future-work}~FW2.)\par

\textbf{L3. Collusion vulnerability and calibration limitations.} A single colluding actor can inflate $\gamma(C)$ to $0.99$ through repeated self-validation (E14), regardless of the aggregation variant: for $\gamma_{\text{default}}$ the last \textsc{Validate} prevails; for $\gamma_{\text{agg}}$ the per-actor maximum with $N_{\text{vals}} = 1$ yields $c_{\text{base}} = 0.99$; and for $\gamma_{\text{cal}}$ self-reported accuracy scores do not constrain a lone validator. The codebase implements four calibration strategies (linear, EWMA, epistemic context, and per-band correction; \S\ref{subsec:calibration}), yet they all lack game-theoretic guarantees: validators face no economic cost for overconfidence or collusion, and accuracy scores are also self-reported. A minimum-validator threshold $N_{\min} \geq 2$ (Lemma~\ref{lem:threshold-mitigation}) prevents a single actor from triggering \textit{validated} status, but does not address Sybil attacks (a single actor creating $N_{\min}$ pseudonymous identities) or coordinated confidence inflation. Full mitigation requires authenticated identities (FW4) and a staking-based mechanism (FW9).
\textbf{L4. Cryptographic authentication gap.} The current implementation relies on self-declared string identifiers, lacking Linked Data Proofs, DIDs, or Byzantine fault tolerance. However, a minimal \texttt{did:key} layer is already implemented (\S\ref{subsec:trust-model}): events can carry Ed25519 signatures, and \texttt{verify\_signature()} validates both DID-derived and registry-derived keys. This is a Phase 1.5 pre-implementation, not a full production authentication system. Full production-grade comparison with nanopublications (Trusty URIs) and PROV-O (LD-Proofs) requires Phase 2 authentication (FW4). (See~\S\ref{subsec:future-work}~FW4.)\par
\textbf{L5. Absence of expert ground truth in synthetic calibration.} The AML evaluation mixes real transaction data with synthetically injected patterns, lacking expert labels or human annotator ratings; the E6 pipeline is explicitly characterized as $96.8$\% \texttt{REGISTER} events from real data and $3.2$\% synthetic governance events. (See~\S\ref{subsec:future-work}~FW5.)

\textbf{L6. Unstructured agent communication.} L2 Markdown in the current implementation imposes no structural constraints on agent reasoning narratives. The current codebase introduces template-governed sections ([Observation], [Reasoning], [Conclusion]) with Pydantic validation (\texttt{l2\_template.py}), which was evaluated in E20 (template effectiveness) but yielded only modest gains in structured reasoning. Full SHACL validation and epistemic context learning are planned for future work (FW6).\par
\textbf{L7. Brittle capability discovery.} \texttt{DataImporter.discover\_classes()} relies solely on \texttt{\_id} suffix matching, which is brittle, coverage-limited, and domain-dependent. (See~\S\ref{subsec:future-work}~FW7.)\par
\textbf{L8. Incomplete Semantic Web interoperability.} While OWL~2 DL bidirectional import/export and RDF-star/SPARQL-star export are already implemented (\S\ref{subsec:semantic-web-interop}), three standard Semantic Web pathways remain unimplemented: SHACL validation for machine-checkable document structure, PROV-O provenance mapping for W3C-compliant lifecycle provenance, and JSON-LD serialization for lightweight web API integration. This limits interoperability with standard Semantic Web toolchains that rely on these formats. (See~\S\ref{subsec:future-work}~FW8.)

\textbf{L9. Absence of an incentive-compatible validation mechanism.} The $\gamma(C)$ function aggregates self-reported confidence values without a staking or slashing mechanism to incentivize truthful reporting; validators face no economic cost for overconfidence or collusion. Basic linear calibration (\texttt{calibration.py}) weights confidence by per-actor accuracy, yet it lacks the game-theoretic guarantees of a full ``Ontological Assertion Market.'' (See~\S\ref{subsec:future-work}~FW9.)\par
\textbf{L10. Informal theorem proofs.} The seven theorems in Section~\ref{subsec:formal-semantics} are proved by rigorous natural-language argument, not by machine-checked derivations in Coq or TLA+. A randomized trace checker (Experiment E24, 10,000 synthetic traces of length 2--100) validates the theorems empirically with 100\% pass rate, providing an independent property-based validation that increases confidence in the natural-language arguments. Machine-checked proofs for unbounded chains remain future work (FW10).\par

\textbf{L11. Overstatement risk in governance claims.} The abstract and introduction have been carefully scoped to the collaborative non-Byzantine trust model. However, readers may still overgeneralize the tamper-evidence guarantees to adversarial settings. We explicitly state that the SHA-256 chain detects tampering post-hoc but does not prevent it, and that actor identity is self-declared in Phase~1. All governance claims are scoped to the collaborative-audit setting; adversarial robustness is planned for Phase~3 (FW4).\par
\textbf{L12. Brittle fork resolution (last-write-wins).} The current fork resolution uses LWW with the total order $\prec$ (Equation~6). While deterministic, LWW is semantically brittle: the merged result depends entirely on timestamp ordering, which may not reflect epistemic intent (e.g., a later \textsc{Deprecate} from a low-accuracy actor overriding an earlier \textsc{Validate} from a high-accuracy actor). The calibration layer ($\gamma_{\text{cal}}$) can weight validators by accuracy, but LWW itself does not consider semantic weight. CRDT-style convergence with semantic merge policies (e.g., quorum-based resolution, accuracy-weighted ordering) is planned for Phase~3 (FW4).\par

\subsection{Complementary Systems and the Agent Governance Landscape}\label{subsec:complementary-systems}

ADL Lite occupies a distinct layer in the broader agent governance landscape. KYA addresses permissions; AgentSafe addresses architecture safety; Talukdar et al.\ address ontology production; Agent Traces addresses provenance; SafeAgent addresses safety protocol. SafeHarness~\cite{safeharness2026} integrates four defense layers into the agent execution lifecycle, reducing unsafe behaviour by $\approx$38\% and attack success by $\approx$42\%, yet it targets execution-time safety rather than capability registration. OpenClaw PRISM~\cite{openclaw_prism2026} provides a zero-fork runtime security layer with ten lifecycle hooks and a tamper-evident audit plane, governing tool invocation and outbound messaging at the gateway level. ADL Lite addresses the question ``How do we record, validate, and govern capabilities?'' (lifecycle registry). These layers are complementary rather than competing: SafeHarness and PRISM consume capability metadata, whereas ADL Lite produces it. A complete ecosystem would employ KYA for permissions, AgentSafe for architecture safety, SafeHarness/PRISM for runtime protection, and ADL Lite for capability lifecycle governance. The research contribution of ADL Lite is to render the capability layer explicit, auditable, and event-first.

\textbf{Authoring friction.} Nanopublications require Turtle/RDF syntax and a publishing pipeline; PROV-O requires RDF/SPARQL tooling; Git signing requires only Git but lacks structured capability metadata. ADL Lite uses Markdown with YAML front matter, which is already the authoring format for most agent documentation.

\textbf{Verification cost.} Nanopublication verification involves resolving Trusty URIs and checking signatures against a key registry; PROV-O verification requires SPARQL queries over a triple store; Git verification checks SHA-1 tree hashes and GPG signatures; ADL Lite verifies SHA-256 hashes and scans the chain in $O(n)$ time ($\approx$1 $\mu$s per event on modest hardware, E13).

\textbf{Storage overhead.} Nanopublications carry full RDF triples and metadata (typically 500--2000 bytes per assertion); PROV-O requires triple-store overhead; Git commits include tree and parent references ($\approx$200 bytes minimum per commit); ADL Lite events are compact JSON objects with SHA-256 hashes ($\approx$150 bytes per event, including hash and linkage fields).

\textbf{Lifecycle expressivity.} ADL Lite provides a total transition matrix governing all transitions between the five lifecycle states (provisional, validated, deprecated, forked, archived) with precondition rules for each transition. None of the baselines provide lifecycle state machines: nanopublications are static assertions; PROV-O records provenance but does not govern status transitions; Git tracks versions but does not distinguish validate/deprecate/fork semantics.

\textbf{Authentication.} Nanopublications use Trusty URIs and RSA/ECDSA signatures; Git uses GPG or SSH signing; ADL Lite uses Ed25519 signatures with optional \texttt{did:key} resolution (\S\ref{subsec:trust-model}). All three provide tamper-evidence but differ in identity binding: GPG keys are self-asserted, Trusty URIs are anchored by publishers, and \texttt{did:key} embeds the public key in the identifier itself.

Several industry and standards initiatives are currently defining agent registry and communication protocols. The \emph{Model Context Protocol} (MCP, Anthropic) provides a standardized way for agents to expose capabilities to LLM context windows; ADL Lite could serve as the \emph{governance backend} for MCP servers, recording which capabilities were registered, validated, and deprecated. The \emph{Agent-to-Agent} (A2A, Google) protocol focuses on inter-agent communication and task delegation; ADL Lite EventChains could embed A2A message traces as \texttt{EVIDENCE} events, providing auditable provenance for delegated tasks. The \emph{AGNTCY} (Linux Foundation) and \emph{NANDA} (NANDA) standards define agent identity and capability ontologies; ADL Lite's L3 predicates could be aligned with their relation vocabularies (e.g., mapping \texttt{isomorphic-to} to AGNTCY equivalence relations). Microsoft \emph{Entra} provides enterprise identity and permissions; ADL Lite could use Entra-issued Verifiable Credentials as the identity layer for validators (planned for a future release). These integrations are not yet implemented; they are identified as high-priority future work (FW13).

\subsection{Positioning and Distinctive Properties}\label{subsec:positioning}

ADL Lite uniquely combines four properties that no existing approach integrates: (a)~document-native authoring in plain Markdown, (b)~built-in tamper detection through cryptographic hash chaining, (c)~lifecycle state machines with declarative preconditions, and (d)~pip-installable deployment requiring no enterprise infrastructure. This enables multiple LLM agents to propose, challenge, and refine concepts within a shared document-native ontology, with the EventChain providing the audit trail and consensus mechanism that renders decentralized authorship accountable.
