This appendix presents the OWL~2 DL alignment fragment that formalizes the ontological dependence patterns from \S\ref{subsec:ontological-dependence}. The fragment declares core category axioms, L3 relation axioms, and L4 action axioms, together with competency questions that demonstrate the intended scope.

\subsection{Core Category Axioms}

\begin{itemize}[leftmargin=2em]
    \item \texttt{adl:Event} $\sqsubseteq$ \texttt{bfo:occurrent} --- events are occurrents in BFO terms.
    \item \texttt{adl:EventChain} $\sqsubseteq$ \texttt{bfo:process} --- an event chain is a process with temporal parts.
    \item \texttt{adl:follows} links temporal parts within a process.
    \item \texttt{adl:Concept} $\sqsubseteq$ \texttt{bfo:generically\_dependent\_continuant} --- concepts are GDCs.
    \item \texttt{adl:Concept} is concretized by an \texttt{EventChain\_Record} (an ICE).
    \item \texttt{adl:hasPreviousHash} and \texttt{adl:hasSHA256Hash} are datatype properties linking events to their cryptographic predecessors.
\end{itemize}

\subsection{L3 Relation Axioms}

The following axioms encode ADL Lite's closed predicate set as OWL object properties with domain/range constraints and inverse declarations.

\begin{verbatim}
adl:isomorphic-to rdf:type owl:ObjectProperty ;
    rdfs:domain adl:Concept ;
    rdfs:range adl:Concept ;
    owl:symmetric true ;
    rdfs:comment "Cross-domain structural alignment" .

adl:specialisation-of rdf:type owl:ObjectProperty ;
    rdfs:domain adl:Concept ;
    rdfs:range adl:Concept ;
    rdfs:subPropertyOf adl:related-to ;
    rdfs:comment "Narrower instance or subtype of target concept" .

adl:fork-of rdf:type owl:ObjectProperty ;
    rdfs:domain adl:Concept ;
    rdfs:range adl:Concept ;
    rdfs:comment "Lineage link from fork to original concept" .

adl:mitigated-by rdf:type owl:ObjectProperty ;
    rdfs:domain adl:Concept ;
    rdfs:range adl:Concept ;
    rdfs:comment "Mitigation or remediation relationship" .
\end{verbatim}

\subsection{L4 Action Axioms (OWL~2 DL Limitation)}

L4 actions (\texttt{REGISTER}, \texttt{VALIDATE}, \texttt{FORK}, etc.) are \emph{intentionally excluded} from the OWL~2 DL fragment because they require temporal reasoning and preconditions that exceed OWL~2 DL expressivity. For example, \texttt{VALIDATE} requires that the concept's current status be \texttt{provisional}, which is a fluent that changes over time. OWL~2 DL cannot express temporal precedence or preconditions over derived state. These are encoded in the YAML ontology registry (\texttt{adl\_core\_ontology.yaml}) and enforced by the ActionExecutor at runtime.

\subsection{Competency Questions}

The following competency questions define the scope of the OWL~2 DL fragment.

\begin{enumerate}[label=CQ\arabic*:,leftmargin=3em]
    \item Which BFO category does an ADL Lite Event belong to? \\ ($\Rightarrow$ answered by \texttt{adl:Event $\sqsubseteq$ bfo:occurrent})
    \item Which BFO category does an ADL Lite Concept belong to? \\ ($\Rightarrow$ answered by \texttt{adl:Concept $\sqsubseteq$ bfo:GDC})
    \item Is the \texttt{isomorphic-to} relation symmetric? \\ ($\Rightarrow$ answered by \texttt{owl:symmetric true})
    \item Is the \texttt{specialisation-of} relation transitive? \\ ($\Rightarrow$ \texttt{owl:TransitiveProperty} not declared; transitivity is a domain-level property, not an ontological one, and is left to the application layer to enforce)
    \item Can a concept be related to itself? \\ ($\Rightarrow$ not constrained in the fragment; application-level validation rules prevent self-relation)
\end{enumerate}

\textbf{Aspects intentionally outside OWL~2 DL expressivity.} (i)~Temporal reasoning: the \texttt{HoldsAt} predicate and the \texttt{Revoked} condition require interval-based semantics that exceed DL-Lite$_\mathcal{R}$. (ii)~Derived state: $\delta(C)$ and $\gamma(C)$ aggregate over event sequences; this requires recursion or fixed-point computation, which is not expressible in OWL~2 DL. (iii)~Preconditions: the ActionExecutor's comparator rules are ground atomic comparisons without quantification or implication; they are not Horn clauses (no head--body implication structure) and cannot be encoded as DL axioms. (iv)~Cryptographic integrity: SHA-256 hash correctness is a computational property, not a logical entailment. These aspects are enforced by the Python runtime, not by the OWL reasoner.

\textbf{SHACL usage.} SHACL shapes would govern L1 schema conformance (e.g., \texttt{adl\_type} must be one of \{discovery, concept, relation, evidence, formal\_seal\}) and L2 template structure (e.g., \texttt{[Observation]}, \texttt{[Reasoning]}, \texttt{[Conclusion]} sections are required). SHACL is not used for L3 relation validation or L4 action preconditions, which are handled by the ActionExecutor. Full SHACL integration is future work (FW6).

The complete OWL~2 DL file is provided as supplementary material (available in the repository's \texttt{supplementary/} directory), where it has been syntax-checked but not yet validated with ROBOT consistency checking. It is intended as a \emph{conceptual alignment anchor}---a starting point for interoperability with BFO-based tools. Full bidirectional import/export (Turtle and RDF/XML) is implemented (\texttt{owl\_export.py} / \texttt{owl\_import.py}); automated BFO/DOLCE/UFO alignment and ROBOT validation are deferred to future work (FW1).
