## 复盘原则

### 无指责文化（Blameless Culture）

1. **不找替罪羊**：永远不问"谁搞砸了？"，而是问"系统为什么允许这个错误发生？"
2. **聚焦系统而非个人**：好的工程师在糟糕的系统里也会犯错 — 修复系统而非指责人
3. **心理安全第一**：害怕被指责的工程师会隐藏问题而非上报问题
4. **每个故障都是礼物**：它暴露了系统的弱点，给了我们在出事前修复的机会
5. **假设善意**：每个决策在当时的背景下都是合理的，复盘是为了理解而非审判

### 禁止的表述 vs 推荐的表述

| ❌ 禁止 | ✅ 推荐 |
|--------|--------|
| "张三配置错了" | "配置校验系统未能拦截非法配置" |
| "李四没有 review 到" | "Code review 流程缺少对配置变更的检查项" |
| "运维搞错命令了" | "该操作缺少二次确认和权限控制" |
| "测试漏掉了这个 case" | "该场景缺少自动化测试覆盖" |
| "值班的人回复太慢了" | "告警系统误报率高，导致告警疲劳" |

## 四、根因分析

### 4.1 故障链路

用自然语言描述故障发生的完整链路：

```
[触发事件]
  → [链式反应步骤 1]
  → [链式反应步骤 2]
  → [用户可见影响]
```

### 4.2 三层原因分析

| 层次 | 描述 | 案例 |
|------|------|------|
| **直接原因** | 触发故障的直接操作或事件 | 部署了一个包含错误超时配置的版本 |
| **深层原因** | 为什么直接原因能够导致故障 | 配置变更无自动化校验，超时参数可设任意值 |
| **系统原因** | 组织/流程层面的根本问题 | 缺少配置变更的 pre-commit hook 和部署前冒烟测试 |

### 4.3 五问法（5 Whys）

1. **为什么服务不可用？**
   → [回答]

2. **为什么 [回答 1] 会发生？**
   → [回答]

3. **为什么 [回答 2] 会发生？**
   → [回答]

4. **为什么 [回答 3] 会发生？**
   → [回答]

5. **为什么 [回答 4] 会发生？**
   → [**根因 — 系统/流程层面的问题**]

## 七、改进措施（Action Items）

> 使用 SMART 原则：具体、可衡量、有 owner、有 deadline。

| ID | 改进措施 | 类型 | 优先级 | 负责人 | 截止日期 | 状态 |
|----|---------|------|--------|--------|---------|------|
| AI-1 | [具体措施] | [预防/检测/恢复/流程] | P0/P1/P2 | @姓名 | YYYY-MM-DD | 未开始 |
| AI-2 | [具体措施] | [类型] | [优先级] | @姓名 | YYYY-MM-DD | 进行中 |

**优先级说明：**
- **P0**：必须立即处理，阻断后续发布
- **P1**：本迭代内完成，高优先级
- **P2**：纳入下个迭代，重要但不紧急

**类型说明：**
- **预防**：防止同类故障再次发生
- **检测**：更快发现故障
- **恢复**：缩短恢复时间
- **流程**：改进团队工作流程

## 复盘会议流程

### 会前准备（主持人）
- [ ] 完成复盘报告草稿（至少填好时间线和根因分析）
- [ ] 收集故障期间的聊天记录、监控截图
- [ ] 确定参会人员：IC、TL、CL、相关服务 owner
- [ ] 预定会议室，45-60 分钟
- [ ] 提前 30 分钟发送报告草稿给参会者

### 会议议程（45 分钟）
| 时间 | 环节 | 内容 |
|------|------|------|
| 0-5 min | 开场 | 说明会议目的：学习而非指责 |
| 5-15 min | 时间线回顾 | 逐条过时间线，补充缺失信息 |
| 15-25 min | 根因讨论 | 讨论深层原因和系统原因 |
| 25-35 min | 改进讨论 | 头脑风暴改进措施 |
| 35-45 min | Action Items | 确定负责人和截止日期 |

### 会后跟进
- [ ] 更新复盘报告最终版
- [ ] 将 Action Items 录入项目管理工具
- [ ] 更新相关 Runbook
- [ ] 在下一次团队周会上同步关键经验
- [ ] 2 周后检查 Action Items 进展

## 复盘质量自检

| 检查项 | 是/否 |
|--------|-------|
| 时间线精确到分钟，每个时间点有操作人 | |
| 根因分析达到了系统/流程层面，不是停在个人操作层面 | |
| 至少运用了一种根因分析方法（5 Whys / 故障树） | |
| 每个 Action Item 有明确 owner 和 deadline | |
| 没有使用"XX 操作失误""XX 忘记 XX"等指责性表述 | |
| 改进措施可验证（做完后能明确判断是否完成） | |
| 没有将"加强培训""提高意识"作为唯一的改进措施 | |
