## 📋 体系总览

本体系覆盖故障全生命周期管理，从检测、响应、通报到复盘改进的完整闭环。

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  检测     │ → │  响应     │ → │  恢复     │ → │  复盘     │
│ Detection │    │ Response │    │ Recovery │    │ Review   │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
     ↑                                               │
     └─────────────── 持续改进 ──────────────────────┘
```

## 🔄 故障处理标准流程

### 角色定义

| 角色 | 职责 | 谁担任 |
|------|------|--------|
| **故障指挥官 (IC)** | 决策拍板、时间线管理、资源协调 | 当值 on-call 主值班 |
| **技术负责人 (TL)** | 故障诊断、修复方案执行 | 相关服务 owner 或 on-call 副值班 |
| **沟通负责人 (CL)** | 对内对外通报、状态页更新 | 轮值或指定人员 |
| **记录员 (Scribe)** | 实时记录时间线和操作 | 故障声明后指定 |

### 四阶段流程

#### 阶段一：检测与声明（0-5 分钟）
1. 监控告警触发或用户反馈到达
2. 确认是否为真实故障（排除误报）
3. 根据[定级框架](./01-severity-framework.md)确定故障级别
4. 在故障频道声明故障：级别、影响范围、IC 是谁
5. 分配角色：IC、TL、CL、Scribe

#### 阶段二：响应与诊断（5 分钟 - N 小时）
1. TL 使用 [Runbook](./02-runbook-template.md) 进行诊断
2. 每 15 分钟（SEV1）或 30 分钟（SEV2）发送状态更新
3. 假设验证时间盒：每条调查路径不超过 15 分钟
4. Scribe 实时记录所有操作和时间点
5. 达到升级条件时自动升级

#### 阶段三：恢复与验证
1. 止血优先：回滚、扩容、降级、切流
2. 指标验证恢复：不只看"看起来好了"，用数据说话
3. 观察 15-30 分钟确保修复稳定
4. 宣布故障解除，发送解除通报
5. 安排复盘会议（48 小时内）

#### 阶段四：复盘与改进
1. 使用[复盘模板](./04-postmortem-template.md)撰写复盘报告
2. 召集无指责复盘会议
3. 产出 Action Items，明确 owner 和 deadline
4. 跟踪 Action Items 闭环
5. 更新 Runbook、告警规则、架构文档

## ⏱️ 关键时间承诺

| 指标 | SEV1 目标 | SEV2 目标 |
|------|----------|----------|
| 检测时间 (MTTD) | < 5 分钟 | < 15 分钟 |
| 响应确认 | < 5 分钟 | < 15 分钟 |
| 恢复时间 (MTTR) | < 30 分钟 | < 2 小时 |
| 复盘完成 | < 48 小时 | < 72 小时 |
| Action Items 闭环率 | ≥ 90% | ≥ 85% |
