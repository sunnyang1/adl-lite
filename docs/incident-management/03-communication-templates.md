# 03 — 沟通通报模板

> 版本：v1.0 | 适用渠道：Slack / 企业微信 / 邮件 / 状态页

---

## 通报原则

1. **宁可过度沟通，不要信息黑洞** — "暂无进展，仍在排查" 也是有效更新
2. **按受众分层** — 工程师需要的细节 ≠ 业务方需要的总结 ≠ 高管需要的判断
3. **固定节奏** — SEV1 每 15 分钟，SEV2 每 30 分钟，即使没有新进展也要更新
4. **单一信息源** — #incidents 频道是唯一真相来源，禁止私下传话
5. **先通报再修复** — 不要在"修好再说"的心态下延迟通报

## 一、内部故障频道通报模板

### 1.1 故障声明（SEV1/SEV2 发生后 5 分钟内）

```
🚨 故障声明

级别：SEV[1/2]
状态：🔴 调查中
服务：[服务名称]
影响：[X]% 用户 [具体症状]，[预计影响描述]

IC（指挥官）：@姓名
TL（技术负责人）：@姓名
CL（沟通负责人）：@姓名
Scribe（记录员）：@姓名

开始时间：HH:MM UTC+8
下次更新：HH:MM UTC+8（约 15 分钟后）

当前已知：[简要描述目前掌握的信息]
```

### 1.2 状态更新（SEV1 每 15 分钟，SEV2 每 30 分钟）

**有进展时：**
```
🔄 故障更新 [第 N 次]

级别：SEV[1/2]
状态：🟡 已定位 / 🔵 修复中
时长：[已持续 X 分钟]

根因假设：[已确认/疑似原因]
已采取措施：
  - [操作 1：时间，结果]
  - [操作 2：时间，结果]

下一步：[计划中的操作]
预计恢复：[ETR 预估]

下次更新：HH:MM
```

**暂无进展时：**
```
🔄 故障更新 [第 N 次]

级别：SEV[1/2]
状态：🔴 仍在调查中
时长：[已持续 X 分钟]

已排除的可能性：
  - [排除项 1]
  - [排除项 2]

当前调查方向：[正在排查的方向]
是否需要支援：[是/否，需要什么帮助]

下次更新：HH:MM
```

### 1.3 故障解除通报

```
✅ 故障解除

级别：SEV[1/2]
服务：[服务名称]
状态：🟢 已恢复

时间线：
  开始：HH:MM
  定位：HH:MM
  修复：HH:MM
  恢复：HH:MM
  总持续：[X 小时 X 分钟]

根因：[简要说明根因]
修复措施：[修复方式]

影响总结：
  - 受影响用户：[数量/比例]
  - 影响范围：[具体描述]
  - 数据影响：[有/无]

后续：
  - 复盘会议：[日期时间]
  - 复盘文档：[链接]
  - 继续监控 [X] 小时

感谢大家在故障期间的协作 🤝
```

## 二、对外通报模板

### 2.1 状态页更新 — 故障通告

**标题**：服务异常通告 — [服务名称]

**正文：**
> 我们正在调查 [服务名称] 的一个异常问题。
>
> **影响**：部分用户可能遇到 [症状描述]。
> **开始时间**：YYYY-MM-DD HH:MM (UTC+8)
> **当前状态**：调查中
>
> 我们会在 [15/30] 分钟内更新进展。
> 如有疑问请联系 support@[company].com。

### 2.2 状态页更新 — 进展更新

**标题**：服务异常更新 — [服务名称]

**正文：**
> 关于 [服务名称] 异常的更新：
>
> **当前状态**：[已定位/修复中]
> **已确认原因**：[简要说明]
> **修复措施**：[正在执行的措施]
> **预计恢复时间**：[如有]
>
> 下次更新：[时间]

### 2.3 状态页更新 — 故障恢复

**标题**：服务恢复正常 — [服务名称]

**正文：**
> [服务名称] 已恢复正常运行。
>
> **故障时间**：YYYY-MM-DD HH:MM 至 HH:MM（持续 [X 小时 X 分钟]）
> **故障原因**：[简要说明]
> **修复措施**：[采取的措施]
>
> 我们正在准备详细的故障复盘报告，将在 48 小时内完成。
> 对此造成的不便深表歉意。

## 三、不同受众的通报要点

### 面向工程团队（#incidents 频道）
- 技术细节完整
- 包含具体的命令、日志、指标
- 讨论假设和排查方向
- 可以要求具体人员支援

### 面向业务/产品团队
- 说明用户影响（多少人、什么功能、多久）
- 预计恢复时间
- 是否需要业务侧配合（如暂停运营活动）
- 用户咨询应如何回复的标准话术

### 面向高管/管理层
模板：
```
📊 故障高管简报

🔴 当前状态：SEV[1/2]，[已持续 X 分钟]

一句话：核心 [服务] [异常症状]，影响约 [X]% 用户

关键判断：
  - 恢复时间预估：[X 分钟内] / [暂无法预估]
  - 是否需要升级：是 / 否
  - 是否需要对外公告：是 / 否
  - 是否有合规/法务风险：是 / 否

当前行动：[2-3 条关键举措]

下次更新：[时间]
```

### 面向客服/运营团队
模板：
```
📞 客服应对指引

故障概要：[一句话说明]
影响范围：[哪些用户/功能]
用户会看到：[具体现象]

标准回复话术：
"我们注意到 [服务] 出现了临时异常，技术团队正在紧急修复中，
  预计 [时间] 内恢复。给您带来不便，深表歉意。"

FAQ：
Q: 用户数据会丢失吗？A: [是/否/不确定]
Q: 需要用户做什么？A: [什么都不需要 / 需要刷新 / 需要重新登录]
Q: 退款/赔偿怎么处理？A: [按标准流程 / 等通知]
```

## 四、通报渠道对照表

| 渠道 | 适用场景 | 负责人 | 频率 |
|------|---------|--------|------|
| #incidents（Slack/企微） | 内部实时协调 | IC | 按级别频率 |
| 状态页（Statuspage） | 对外用户通告 | CL | 每个阶段一次 |
| 邮件 | 客户通知、高管简报 | CL | 故障开始 + 故障解除 |
| 客服群 | 客服话术同步 | CL | 故障开始 + 恢复 |
| 全员公告 | 重大故障（SEV1>30min） | CL/VP Eng | 根据影响决定 |

## 五、通报检查清单

故障处理过程中，CL 需确认：

- [ ] 故障声明已发送（5 分钟内）
- [ ] 状态页已更新（SEV1/SEV2）
- [ ] 客服团队已通知（影响用户时）
- [ ] 高管已通知（SEV1 立即，SEV2 超过 30 分钟时）
- [ ] 每次更新都有发送（按频率要求）
- [ ] 故障解除通报已发送
- [ ] 状态页已标记恢复
- [ ] 复盘会议已安排
