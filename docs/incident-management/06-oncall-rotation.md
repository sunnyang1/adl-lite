# 06 — On-Call 值班制度设计

> 版本：v1.0 | 目标：平衡系统可靠性与工程师健康

---

## 一、值班制度设计原则

### 核心原则
1. **可持续性优先** — 没有人应该长期在高压下值班
2. **知识覆盖** — 每个服务至少有 2 人掌握，杜绝单点知识
3. **权责对等** — 值班工程师有权做出紧急决策，无需层层审批
4. **合理补偿** — 值班应该有对应的薪酬或调休补偿
5. **告警质量 > 告警数量** — 宁可漏报一个，不可每天误报十个

## 二、排班设计

### 2.1 基本参数

```yaml
rotation:
  name: "[团队名]-primary"
  timezone: "Asia/Shanghai"  # UTC+8
  rotation_type: weekly      # 周轮换
  handoff_time: "10:00"      # 上午交班（绝不在半夜交班）
  handoff_day: monday        # 周一交班

participants:
  min_rotation_size: 4       # 最少 4 人轮换，防止 burnout
  max_consecutive_weeks: 2   # 连续值班不超过 2 周
  shadow_period_weeks: 2     # 新人影子值班 2 周后才能独立
  
  # 排班节奏
  # 4 人团队：每人每 4 周值班 1 周 = 25% 时间
  # 6 人团队：每人每 6 周值班 1 周 = 17% 时间
  # 8 人团队：每人每 8 周值班 1 周 = 12.5% 时间
```

### 2.2 分层值班体系

```
                    ┌──────────────┐
                    │  Tier 1 升级  │  VP Eng / CTO
                    │  SEV1 超过    │  任何时候可联系
                    │  30 分钟触发  │
                    ├──────────────┤
                    │  Tier 2 升级  │  技术经理
                    │  值班 15 分钟  │  工作时间
                    │  未响应触发    │
                    ├──────────────┤
   ┌──────────────────────────────────┐
   │         主值班 (Primary)          │  响应所有告警
   │         副值班 (Secondary)        │  主值班无响应时接替
   │         影子值班 (Shadow)         │  学习为主，不直接处理
   └──────────────────────────────────┘
```

### 2.3 升级策略

```yaml
escalation_policy:
  - level: 1
    target: "主值班 (Primary)"
    timeout: 5_minutes       # 5 分钟不确认 → 升级
    notification: "电话 + 企微 + 短信"
    
  - level: 2
    target: "副值班 (Secondary)"
    timeout: 10_minutes
    
  - level: 3
    target: "技术经理"
    timeout: 15_minutes
    condition: "SEV1 或 SEV2 持续超过 30 分钟"
    
  - level: 4
    target: "VP Engineering"
    timeout: 0               # 立即通知
    condition: "SEV1 持续超过 30 分钟 OR 数据安全问题"
```

## 三、值班交接流程

### 3.1 交接清单

**交班方（上一班）需要：**
- [ ] 回顾过去一周的所有故障和告警
- [ ] 列出未解决的 Action Items
- [ ] 标注高风险变更（即将发生的部署、配置变更）
- [ ] 更新 Runbook 变更（如有新发现或修改）
- [ ] 确认监控大盘和告警均正常
- [ ] 在 #oncall-handoff 频道发送交接记录

**接班方（当前班）需要：**
- [ ] 阅读交接记录
- [ ] 确认能收到告警通知（测试页验证）
- [ ] 了解本周计划中的变更
- [ ] 确认各服务监控面板可正常访问
- [ ] 在交接记录中回复确认

### 3.2 交接记录模板

```markdown
## 🔄 On-Call 交接 — W[周数] (YYYY-MM-DD)

**交班人**：@姓名 → **接班人**：@姓名

### 本周故障总结
| 日期 | 级别 | 服务 | 简述 | 状态 |
|------|------|------|------|------|
| MM-DD | SEV2 | payment | Redis 连接池耗尽 | 已解决，待复盘 |

### 未解决的 Action Items
| ID | 内容 | Owner | Deadline |
|----|------|-------|----------|
| AI-1 | 增加连接池监控 | @张三 | MM-DD |

### 本周计划变更
| 日期 | 变更内容 | 风险等级 | 负责人 |
|------|---------|---------|--------|
| MM-DD | payment 服务升级 v2.3 | 中 | @李四 |

### 注意事项
- [任何需要接班方特别关注的事项]

### 接班人确认
- [ ] 告警通知测试通过
- [ ] 监控面板可访问
- [ ] 已了解本周变更计划
```

## 四、值班工程师权责

### 值班工程师有权：
1. **紧急回滚** — 无需审批即可回滚任何部署
2. **临时降级** — 关闭非核心功能保护核心链路
3. **扩容** — 在合理范围内临时扩容
4. **切流** — 按预案将流量切到备用集群
5. **要求支援** — 任何时候可以要求任何人协助

### 值班工程师有责：
1. **及时响应** — SEV1 5 分钟、SEV2 15 分钟内确认告警
2. **规范操作** — 所有操作必须在故障频道记录
3. **升级上报** — 超出能力范围时及时升级
4. **善后跟进** — 发起复盘、更新 Runbook
5. **知会交接** — 确保交接信息完整

## 五、值班健康度监控

### 关键指标

| 指标 | 健康阈值 | 警戒线 | 需要行动 |
|------|---------|--------|---------|
| **每周告警数（人均）** | < 5 次 | 5-10 次 | > 10 次 |
| **误报率** | < 20% | 20-40% | > 40% |
| **非工作时间告警占比** | < 30% | 30-50% | > 50% |
| **MTTR（SEV2 以上）** | < 30 min | 30-60 min | > 60 min |
| **值班满意度** | > 4.0/5 | 3.0-4.0 | < 3.0 |

### 告警治理

**告警清理频率**：每月一次

**清理标准：**
- 连续 3 次触发但无需人工介入 → 降级为 warning 或取消
- 2 周内触发超过 20 次 → 检查阈值是否合理
- 触发后值班工程师不知道要做什么 → 补充 Runbook
- 同一类问题反复告警 → 确认之前的修复是否有效

### 值班工程师保护

```yaml
wellbeing_rules:
  # 重大故障后强制休息
  post_sev1_rest:
    condition: "处理 SEV1 超过 2 小时"
    action: "下一个工作日强制休息半天"
    
  # 夜间告警保护
  night_page_protection:
    condition: "夜间（22:00-08:00）被叫醒处理 SEV1/SEV2"
    action: "第二天可以晚到或休息半天"
    
  # 值班频率上限
  max_oncall_load:
    condition: "连续 2 周值班 OR 每月值班超过 1 周"
    action: "必须安排休息周，不参与值班"
```

## 六、补偿方案

### 建议补偿结构

| 类型 | 补偿方式 | 说明 |
|------|---------|------|
| **值班津贴** | 固定金额/周 | 承担值班责任的基础补偿 |
| **非工作时间故障处理** | 按小时计调休 | 22:00-08:00 及周末 |
| **重大故障（SEV1）** | 额外调休半天 | 处理 SEV1 超过 2 小时 |
| **法定节假日值班** | 3 倍调休或津贴 | 按公司政策 |

## 七、新人上手指南

### 影子值班期（2 周）

**第一周：观察**
- [ ] 阅读所有关键服务的 Runbook
- [ ] 熟悉监控面板和告警配置
- [ ] 旁观主值班处理告警
- [ ] 了解故障响应流程

**第二周：影子操作**
- [ ] 主值班指导下处理低级别告警（SEV3/SEV4）
- [ ] 模拟执行 Runbook 中的修复步骤
- [ ] 参与一次复盘会议
- [ ] 完成值班能力评估

### 独立值班前检查清单
- [ ] 能独立完成所有 Runbook 中的修复操作
- [ ] 知道如何升级和寻求帮助
- [ ] 了解各服务的依赖关系
- [ ] 知道哪些操作需要审批、哪些可以直接执行
- [ ] 通过值班能力评估（由技术经理确认）
