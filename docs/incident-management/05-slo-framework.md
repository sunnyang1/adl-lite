# 05 — SLO/SLI 框架与错误预算策略

> 版本：v1.0 | 目标：让告警有意义，让可靠性可衡量

---

## 核心概念

| 术语 | 定义 | 简单解释 |
|------|------|---------|
| **SLI** | Service Level Indicator，服务水平指标 | 具体衡量什么（如请求成功率） |
| **SLO** | Service Level Objective，服务水平目标 | 指标要多好（如 99.95% 成功率） |
| **SLA** | Service Level Agreement，服务水平协议 | 达不到会怎样（通常涉及赔偿） |
| **错误预算** | Error Budget | 允许的不可用时间，用来平衡可靠性和发布速度 |

## 一、SLI 定义模板

### 模板

```yaml
# SLI 定义：每个服务按此模板填写
service: [服务名]
owner: [负责团队]
measurement_window: 30d  # 测量窗口

slis:
  availability:
    description: "有效请求占总请求的比例"
    metric: |
      sum(rate(http_requests_total{service="[服务名]", status!~"5.."}[5m]))
      /
      sum(rate(http_requests_total{service="[服务名]"}[5m]))
    good_event: "HTTP 状态码 < 500"
    valid_event: "所有 HTTP 请求（排除健康检查）"
    
  latency:
    description: "请求在阈值内完成的比例"
    metric: |
      histogram_quantile(0.99,
        sum(rate(http_request_duration_seconds_bucket{service="[服务名]"}[5m])) by (le)
      )
    threshold: "[X]ms at p99"
    
  throughput:
    description: "每秒成功处理请求数"
    metric: |
      sum(rate(http_requests_total{service="[服务名]", status!~"5.."}[5m]))
```

### SLI 选型指南

| 服务类型 | 建议 SLI | 为什么 |
|---------|---------|--------|
| **HTTP API** | 可用性 + 延迟 | 用户最直接感受到的指标 |
| **消息队列消费者** | 消费延迟 + 积压量 | 衡量处理能力 |
| **定时任务** | 成功率 + 执行时间 | 任务是否按时完成 |
| **数据库** | 可用性 + 查询延迟 | 核心依赖的质量 |
| **前端页面** | 加载时间 (LCP) + 错误率 | 用户感知的性能 |
| **数据管道** | 数据新鲜度 + 处理延迟 | 数据是否最新 |

## 二、SLO 定义模板

```yaml
# SLO 定义
service: [服务名]
review_frequency: monthly  # 每月评审

slos:
  - sli: availability
    target: 99.95%          # 对于核心服务
    window: 30d
    error_budget_minutes: 21.6   # 每月允许约 22 分钟不可用
    
  - sli: latency
    target: 99.0%           # p99 延迟在阈值内
    window: 30d
    
  - sli: throughput
    target: [根据容量规划设定]
    window: 30d
```

### 不同服务等级的 SLO 建议值

| 服务等级 | 可用性 SLO | 月度不可用时间 | 延迟 SLO (p99) | 适用场景 |
|---------|-----------|-------------|----------------|---------|
| **Tier 1**（核心） | 99.95% | < 22 分钟 | < 100ms | 支付、登录、核心 API |
| **Tier 2**（重要） | 99.9% | < 43 分钟 | < 250ms | 搜索、推荐、通知 |
| **Tier 3**（一般） | 99.5% | < 3.6 小时 | < 500ms | 报表、后台管理 |
| **Tier 4**（非关键 ）| 99.0% | < 7.2 小时 | < 1s | 内部工具、实验功能 |

## 三、错误预算策略

### 核心原则

> **错误预算 = 1 - SLO**。它是"允许犯错的额度"。
> 预算有剩余 → 可以冒风险（快速迭代）
> 预算耗尽 → 必须保守（优先可靠性）

### 月度错误预算速算表

| SLO | 月度允许不可用时间 |
|-----|-----------------|
| 99.999%（5个9） | 26 秒 |
| 99.99%（4个9） | 4.3 分钟 |
| 99.95%（3.5个9） | 21.6 分钟 |
| 99.9%（3个9） | 43.2 分钟 |
| 99.5%（2.5个9） | 3.6 小时 |
| 99.0%（2个9） | 7.2 小时 |

### 错误预算消耗策略

```
预算剩余 > 50%  ✅ 正常迭代
  → 正常发布节奏，可以承担发布风险
  → 允许进行架构变更和实验

预算剩余 25%-50%  ⚠️ 审慎发布
  → 暂停高风险变更
  → 技术经理评审每次发布
  → 优先修复已知问题

预算剩余 < 25%  🔴 可靠性优先
  → 冻结所有非紧急发布
  → 全员聚焦可靠性改进
  → 每日 review 错误预算消耗

预算耗尽  🚫 紧急状态
  → 完全冻结非关键发布
  → VP Eng 主持评审会议
  → 制定预算恢复计划
```

### 消耗速率告警

```yaml
burn_rate_alerts:
  # 快烧告警：2 小时内耗尽预算 → 立即 page
  - severity: page
    short_window: 5m
    long_window: 1h
    burn_rate: 14.4x
    
  # 中速告警：5 天内耗尽预算 → 创建工单
  - severity: ticket
    short_window: 30m
    long_window: 6h
    burn_rate: 6x
    
  # 慢速告警：30 天内耗尽预算 → 周报提醒
  - severity: report
    short_window: 3h
    long_window: 3d
    burn_rate: 1x
```

## 四、SLO 评审流程

### 月度评审会议（30 分钟）

| 时间 | 议程 |
|------|------|
| 0-5 min | 回顾各服务 SLO 达标情况 |
| 5-10 min | 分析错误预算消耗大户 |
| 10-15 min | 讨论是否需要调整 SLO |
| 15-25 min | 决定下个月的发布策略（基于预算） |
| 25-30 min | Action Items 确认 |

### SLO 调整原则

1. **只能调严格，不能调宽松** — 除非有充分理由
2. **SLO 调整必须经技术经理审批**
3. **SLO 上调后，先观察 30 天再正式生效**
4. **服务等级变更（Tier 调整）需 VP Eng 审批**

## 五、监控仪表盘要求

每个 Tier 1 和 Tier 2 服务必须有以下仪表盘：

### 必备面板

| 面板 | 内容 | 刷新频率 |
|------|------|---------|
| **SLI 总览** | 各 SLI 当前值和 SLO 达标线 | 1 分钟 |
| **错误预算剩余** | 当月剩余错误预算（分钟数和百分比） | 1 分钟 |
| **消耗速率** | 错误预算消耗速率，含告警线 | 5 分钟 |
| **按状态码分布** | 各类 HTTP 状态码的占比趋势 | 1 分钟 |
| **延迟分位数** | p50/p90/p95/p99 延迟趋势 | 1 分钟 |
| **依赖服务 SLI** | 关键依赖服务的健康状态 | 1 分钟 |

## 六、常见问题

### Q: 我们的 SLO 应该设多高？
**A**: 从当前实际水平出发，设一个"略有挑战但可达"的目标。例如，如果当前可用性是 99.8%，先设为 99.9%，稳定 3 个月后再讨论是否提高到 99.95%。**不要一开始就设 99.99%**，那只会训练团队忽略 SLO。

### Q: 错误预算烧完了怎么办？
**A**: 按策略执行可靠性冻结。这不是惩罚，而是信号 — 系统当前的稳定性不足以支持快速迭代，需要先加固。

### Q: SLO 和 SLA 的区别是什么？
**A**: SLO 是内部目标，不达标通常没有直接后果。SLA 是对客户的承诺，不达标可能涉及赔偿。**永远不要让 SLO 等于 SLA** — SLO 应该更严格，作为 SLA 的提前预警。

### Q: 怎么处理计划内维护窗口？
**A**: 计划内维护不计入错误预算。但维护窗口本身也要有预算和时间限制。
