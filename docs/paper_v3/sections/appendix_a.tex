\section{PROV-O Export Example}\label{a.-prov-o-export-example}

This appendix provides a complete, syntactically valid Turtle serialization of an ADL Lite \texttt{EventChain} exported to the W3C PROV-O ontology \footnote{W3C PROV-O: \url{https://www.w3.org/TR/prov-o/}.}. The example uses the \texttt{disc-capital-trap} concept (Scenario 1 from the E5 multi-agent pilot study, Section 4.7), which undergoes the full lifecycle: registration, validation, challenge, and fork. The serialization demonstrates how each ADL event maps to a \texttt{prov:Activity}, how agent roles map to \texttt{prov:Agent}, and how cryptographic hashes and causal links map to PROV-O's provenance relations.

\begin{center}\rule{0.5\linewidth}{0.5pt}\end{center}


\subsection{A.1 Prefix Declarations}\label{a.1-prefix-declarations}

\begin{lstlisting}
@prefix prov: \textless{http://www.w3.org/ns/prov\#\textgreater{} .}
@prefix adl:  \textless{https://adl{-}lite.org/ns/\textgreater{} .}
@prefix xsd:  \textless{http://www.w3.org/2001/XMLSchema\#\textgreater{} .}
@prefix rdfs: \textless{http://www.w3.org/2000/01/rdf{-}schema\#\textgreater{} .}
@prefix foaf: \textless{http://xmlns.com/foaf/0.1/\textgreater{} .}
\end{lstlisting}

The \texttt{adl:} namespace is a placeholder for the ADL Lite vocabulary URI. In a production deployment, this would resolve to the canonical namespace document (e.g., \texttt{https://adl-lite.org/ns/1.0/}). All PROV-O core terms (\texttt{prov:Entity}, \texttt{prov:Activity}, \texttt{prov:Agent}, \texttt{prov:wasGeneratedBy}, \texttt{prov:wasAssociatedWith}, \texttt{prov:wasInformedBy}, \texttt{prov:startedAtTime}) use the standard W3C namespace.

\begin{center}\rule{0.5\linewidth}{0.5pt}\end{center}


\subsection{A.2 Concept Entity}\label{a.2-concept-entity}

The concept itself is modelled as a \texttt{prov:Entity} that is generated by the activities (events) in its chain. The entity carries ADL-specific properties (confidence, status, namespace) as datatype annotations.

\begin{lstlisting}
adl:disc{-capital{-}trap}
    a prov:Entity ;
    rdfs:label "Capital Trap"@en ;
    adl:conceptId "disc{-capital{-}trap" ;}
    adl:namespace "disc" ;
    adl:confidence "0.85"\^{\^{}xsd:float ;}
    adl:status "provisional" ;
    prov:wasGeneratedBy
        adl:evt{-register{-}001 ,}
        adl:evt{-validate{-}001 ,}
        adl:evt{-challenge{-}001 ,}
        adl:evt{-fork{-}001 ;}
    prov:wasAttributedTo adl:actor{-discoverer .}
\end{lstlisting}

The \texttt{prov:wasGeneratedBy} property lists every activity that contributed to the concept's current state. The order of generation is implicit in the activity timestamps (Section A.3) and explicit in the \texttt{adl:previousEvent} links that form the cryptographic chain. The \texttt{prov:wasAttributedTo} property identifies the Discoverer as the original author.

\begin{center}\rule{0.5\linewidth}{0.5pt}\end{center}


\subsection{A.3 Event Activities}\label{a.3-event-activities}

Each event in the chain is modelled as a \texttt{prov:Activity} with its associated agent, timestamp, causal predecessors, and ADL-specific cryptographic metadata.

\subsubsection{Registration Event}\label{a.3.1-registration-event}

The first event in the chain has no cryptographic predecessor; its \texttt{adl:previousEvent} is the literal string \texttt{"genesis"}, indicating the start of the chain.

\begin{lstlisting}
adl:evt{-register{-}001}
    a prov:Activity , adl:RegisterEvent ;
    rdfs:label "Register disc{-capital{-}trap"@en ;}
    prov:wasAssociatedWith adl:actor{-discoverer ;}
    prov:startedAtTime "2024{-01{-}15T09:00:00Z"\^{}\^{}xsd:dateTime ;}
    adl:eventType "REGISTER" ;
    adl:eventIndex "0"\^{\^{}xsd:integer ;}
    adl:previousEvent "genesis" ;
    adl:eventHash "sha256:a1b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef123456" ;
    adl:payloadConfidence "0.75"\^{\^{}xsd:float ;}
    adl:payloadSummary "Discoverer proposes Capital Trap pattern for review" .
\end{lstlisting}

\subsubsection{Validation Event}\label{a.3.2-validation-event}

The validation event is associated with the Reviewer agent. Its \texttt{prov:wasInformedBy} property references the registration event, establishing the causal link in PROV-O terms. The \texttt{adl:previousEvent} property contains the SHA-256 hash of the registration event, enabling cryptographic chain verification.

\begin{lstlisting}
adl:evt{-validate{-}001}
    a prov:Activity , adl:ValidateEvent ;
    rdfs:label "Validate disc{-capital{-}trap"@en ;}
    prov:wasAssociatedWith adl:actor{-reviewer ;}
    prov:wasInformedBy adl:evt{-register{-}001 ;}
    prov:startedAtTime "2024{-01{-}15T14:30:00Z"\^{}\^{}xsd:dateTime ;}
    adl:eventType "VALIDATE" ;
    adl:eventIndex "1"\^{\^{}xsd:integer ;}
    adl:previousEvent "sha256:a1b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef123456" ;
    adl:eventHash "sha256:e5f6a7b8c9d0123456789012345678901234fedcba0987654321fedcba098765" ;
    adl:payloadConfidence "0.85"\^{\^{}xsd:float ;}
    adl:payloadValidator "reviewer{-agent{-}7" ;}
    adl:payloadSummary "Reviewer validates with confidence 0.85" .
\end{lstlisting}

\subsubsection{Challenge Event}\label{a.3.3-challenge-event}

The challenge event is initiated by the Skeptic agent. It represents a contested validation and triggers the fork workflow in Scenario 1.

\begin{lstlisting}
adl:evt{-challenge{-}001}
    a prov:Activity , adl:ChallengeEvent ;
    rdfs:label "Challenge disc{-capital{-}trap validation"@en ;}
    prov:wasAssociatedWith adl:actor{-skeptic ;}
    prov:wasInformedBy adl:evt{-validate{-}001 ;}
    prov:startedAtTime "2024{-01{-}16T10:15:00Z"\^{}\^{}xsd:dateTime ;}
    adl:eventType "CHALLENGE" ;
    adl:eventIndex "2"\^{\^{}xsd:integer ;}
    adl:previousEvent "sha256:e5f6a7b8c9d0123456789012345678901234fedcba0987654321fedcba098765" ;
    adl:eventHash "sha256:1a2b3c4d5e6f789012345678901234567890abcde1234567890abcde1234567" ;
    adl:payloadReason "Confidence threshold insufficient for production use" ;
    adl:payloadSummary "Skeptic challenges validation; proposes fork" .
\end{lstlisting}

\subsubsection{Fork Event}\label{a.3.4-fork-event}

The fork event is initiated by the Merger agent. It creates a new concept (\texttt{disc-capital-trap-v2}) and marks the original concept as \texttt{forked}. The \texttt{adl:forkTarget} property references the new concept entity.

\begin{lstlisting}
adl:evt{-fork{-}001}
    a prov:Activity , adl:ForkEvent ;
    rdfs:label "Fork disc{-capital{-}trap"@en ;}
    prov:wasAssociatedWith adl:actor{-merger ;}
    prov:wasInformedBy adl:evt{-challenge{-}001 ;}
    prov:startedAtTime "2024{-01{-}16T16:00:00Z"\^{}\^{}xsd:dateTime ;}
    adl:eventType "FORK" ;
    adl:eventIndex "3"\^{\^{}xsd:integer ;}
    adl:previousEvent "sha256:1a2b3c4d5e6f789012345678901234567890abcde1234567890abcde1234567" ;
    adl:eventHash "sha256:9a8b7c6d5e4f321098765432109876543210fedcb0987654321fedcb0987654" ;
    adl:forkTarget adl:disc{-capital{-}trap{-}v2 ;}
    adl:payloadSummary "Merger resolves challenge by forking to disc{-capital{-}trap{-}v2" .}
\end{lstlisting}

\subsection{A.4 Agent Definitions}\label{a.4-agent-definitions}

Each agent role in the 5-agent harness is modelled as a \texttt{prov:Agent}. The \texttt{foaf:name} property provides a human-readable label; the \texttt{adl:role} property identifies the agent's function in the ADL workflow.

\begin{lstlisting}
adl:actor{-discoverer}
    a prov:Agent ;
    foaf:name "Discoverer" ;
    adl:role "discoverer" ;
    rdfs:comment "Proposes new concepts for inclusion in the ADL corpus"@en .

adl:actor{-reviewer}
    a prov:Agent ;
    foaf:name "Reviewer" ;
    adl:role "reviewer" ;
    rdfs:comment "Validates proposed concepts against quality criteria"@en .

adl:actor{-skeptic}
    a prov:Agent ;
    foaf:name "Skeptic" ;
    adl:role "skeptic" ;
    rdfs:comment "Challenges validations and proposes forks when criteria are unmet"@en .

adl:actor{-merger}
    a prov:Agent ;
    foaf:name "Merger" ;
    adl:role "merger" ;
    rdfs:comment "Resolves challenges by merging or forking contested concepts"@en .

adl:actor{-librarian}
    a prov:Agent ;
    foaf:name "Librarian" ;
    adl:role "librarian" ;
    rdfs:comment "Archives deprecated or superseded concepts with evidence"@en .
\end{lstlisting}

\subsection{A.5 Fork Target Entity}\label{a.5-fork-target-entity}

The fork event creates a new concept entity. This entity starts its own event chain at \texttt{provisional} status and is independently validated (or not) by the Reviewer agent.

\begin{lstlisting}
adl:disc{-capital{-}trap{-}v2}
    a prov:Entity ;
    rdfs:label "Capital Trap v2"@en ;
    adl:conceptId "disc{-capital{-}trap{-}v2" ;}
    adl:namespace "disc" ;
    adl:confidence "0.0"\^{\^{}xsd:float ;}
    adl:status "provisional" ;
    prov:wasDerivedFrom adl:disc{-capital{-}trap ;}
    prov:wasGeneratedBy adl:evt{-fork{-}001 ;}
    prov:wasAttributedTo adl:actor{-merger .}
\end{lstlisting}

The \texttt{prov:wasDerivedFrom} property records the provenance relationship between the forked concept and its parent. The \texttt{prov:wasGeneratedBy} property references the fork activity that created it. The initial confidence is 0.0 because the forked concept has not yet been independently validated.

\subsection{A.6 Cryptographic Chain Verification}\label{a.6-cryptographic-chain-verification}

The PROV-O export preserves the cryptographic integrity of the original ADL event chain. An external verifier can reconstruct the chain as follows:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Sort the activities by \texttt{adl:eventIndex} to recover event order.
\item
  For each activity, verify that \texttt{adl:previousEvent} equals the \texttt{adl:eventHash} of the preceding activity (or \texttt{"genesis"} for index 0).
\item
  Recompute the SHA-256 digest of each activity's canonical content and confirm it matches \texttt{adl:eventHash}.
\item
  Verify that \texttt{prov:wasInformedBy} relations align with \texttt{adl:previousEvent} links (causal consistency).
\end{enumerate}

This procedure ensures that the PROV-O serialization is not merely a passive export but a verifiable provenance record. Any tampering with the exported Turtle file---e.g., altering an event hash, changing a timestamp, or inserting a spurious \texttt{wasInformedBy} link---can be detected by recomputing the chain digests and comparing them against the stored \texttt{adl:eventHash} values.
